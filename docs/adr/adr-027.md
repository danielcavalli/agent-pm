# ADR 027: Cross-Task Async Commentary for Inter-Agent Communication

## Status

Accepted

## Context

Agents executing stories in parallel operate in isolated sessions with no shared memory or direct communication channel. An agent working on a database migration may discover information relevant to an agent implementing an API layer, but the two agents never interact at the process level. The filesystem could serve as a shared medium for asynchronous annotation, but there was no structured mechanism for agents to leave observations addressed to specific tasks.

Email-style message passing was considered but rejected because it requires sender and receiver to agree on identifiers and implies synchronous delivery expectations. A pub/sub system was rejected because it introduces infrastructure dependencies that conflict with the file-based architecture (ADR-001, ADR-008).

The system also needed to support human-authored comments for cases where a developer wants to leave guidance for an agent without modifying story acceptance criteria.

## Decision

We will implement a cross-task async commentary system stored as individual YAML files in `.pm/comments/`, with a companion index file at `.pm/comments/index.yaml` for efficient lookup.

Each comment is identified by a sequential six-digit code (`C000001`, `C000002`, etc.) and stored in a file named `{id}-{sanitized-content-prefix}.yaml`. Comments target a specific task via `target_task_id`, which must be a valid epic or story code matching `^[A-Z]{2,6}-(E\d{3})(?:-S\d{3})?$`. The target task must be in a targetable status (`backlog`, `in_progress`, or `done`); comments cannot be added to cancelled tasks.

Authors are discriminated by type: agent authors carry an `agent_id`, human authors carry a `name` (defaulting to "anonymous"). The `comment_type` field (`agent` or `human`) enables filtering and differential treatment in the consolidation pipeline.

Each comment carries a `consolidated` flag (initially `false`) and a `consumed_by` array (initially empty). The `consolidated` flag is set to `true` by the consolidation pipeline after ingestion. The `consumed_by` array is populated by agents that have read and acted on the comment. Together, these fields drive garbage collection eligibility (ADR-029): a comment must be consolidated AND either consumed or targeting a completed task before it can be deleted.

The index file maintains two structures: a flat `comments` array containing full comment objects, and a `by_task` map keyed by task ID containing lightweight `CommentIndexEntry` stubs (`{ comment_id, task_reference, created_at }`) for efficient lookup of comments relevant to a specific story or epic. Retrieving full comment data from a `by_task` lookup requires a second read from the individual comment file. The index is updated synchronously on every comment addition.

## Consequences

Agents can now share observations across task boundaries without coupling their execution. An agent that discovers a missing API endpoint while implementing a frontend feature can post a comment targeted at the backend story, then continue its own work. The receiving agent (or a human reviewing the board) sees the comment when it loads the target task's context.

The comment system feeds directly into the consolidation pipeline (ADR-026). Comments are ingested alongside execution reports, providing additional signal for synthesis. Comments flagged as agent-authored with semantic-type observations are candidates for ADR promotion during consolidation.

The individual-file-per-comment design means the `.pm/comments/` directory grows linearly with the number of comments. For active projects with many agents, this could produce hundreds of files. The index file mitigates the read cost, but the write cost (updating the index on every addition) is proportional to the index size. Garbage collection (ADR-029) bounds the growth by deleting consolidated and consumed comments after their TTL expires.

The sequential ID generation scans existing files on every addition to find the next number. This is O(n) in the number of comment files, but the operation is fast for the expected scale (hundreds, not millions, of comments).

The `consumed_by` tracking is a best-effort mechanism. Nothing forces an agent to mark a comment as consumed. Comments from agents that crash or ignore comments will linger until their TTL expires and the target task completes. The GC's dual condition (consumed OR task done) handles this gracefully.

Comment addition does not currently use advisory file locking (ADR-025) when updating the index file. Two agents posting comments simultaneously can race on the index update, potentially losing one entry. The individual comment files are safe (unique filenames prevent collisions), but the index is vulnerable to last-writer-wins corruption under concurrent writes.

Human-authored comments have the same lifecycle as agent-authored ones, including consolidation and GC. This means developer guidance notes will eventually be deleted if the project is active enough to trigger GC. For permanent guidance, story acceptance criteria or ADRs remain the appropriate vehicle.
