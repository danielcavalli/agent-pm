# ADR 008: Four Interfaces to One Data Layer

## Status

Accepted

## Context

pm provides four distinct ways for users and AI agents to interact with project management data: a traditional CLI for direct terminal use, an MCP server for AI agent tool calls, slash commands for guided multi-step workflows, and an interactive TUI dashboard for visual project monitoring. Each interface serves a different audience and interaction pattern, but they all operate on the same underlying data — YAML files stored in `.pm/` at the repository root (see [ADR 024](adr-024.md)).

The fundamental architectural question is how these four interfaces coordinate. In a traditional multi-interface application, you would introduce a service layer or API server that mediates all access to the data store. Clients would communicate with this central service via HTTP, RPC, or some other protocol, and the service would handle locking, transactions, and consistency. This pattern is well-understood and solves real problems: concurrent writes, cache invalidation, event propagation, and access control.

However, pm operates in a fundamentally different environment. There is exactly one user per installation. The tool runs locally on that user's machine. AI agents invoke pm sequentially — they wait for one tool call to complete before issuing the next. The TUI is primarily read-only, with limited writes for agent escalation responses (see [ADR 006](adr-006.md) and [ADR 021](ADR-021.md)). Under these constraints, the complexity of a service layer — managing connections, handling concurrent requests, maintaining process state — provides no benefit and adds operational overhead.

The alternative is to eliminate the coordination layer entirely and let all four interfaces operate directly on the YAML files. The files themselves become the integration point. There is no server to start, no port to configure, no connection to manage, no state to synchronize. Any interface can read the current state by parsing the YAML files, and any interface can modify state by writing them.

## Decision

All four interfaces — CLI, MCP server, slash commands, and TUI — read from and write to the same YAML files in `.pm/` with no intermediary service layer. The YAML files on disk are the single source of truth, and the filesystem is the only integration point between interfaces.

The CLI reads and writes YAML files through the `readYaml`/`writeYaml`/`updateYaml` functions in `src/lib/fs.ts`, validating data against Zod schemas on every read. The MCP server delegates to the CLI via subprocess (see ADR 006), so it inherits the same data access path. Slash commands instruct AI agents to invoke `pm` CLI commands, which again use the same read/write functions. The TUI reads YAML files directly using `fs.watch` with a 300ms debounce to detect changes and re-render.

There is no database, no API server, no message queue, and no lock file. Consistency is maintained by the sequential nature of the tool's usage: one human or one AI agent operates at a time.

## Consequences

The most significant benefit is operational simplicity. There is no daemon to start, no port conflicts to debug, no connection timeouts to handle, and no process to monitor. Running `pm status` works immediately after installation — it just reads files. This is especially valuable for AI agent integration, where the agent needs to invoke tools reliably without managing service lifecycle. The MCP server is stateless; if it restarts, nothing is lost because all state lives in the YAML files.

This architecture also makes the system highly inspectable and debuggable. A user can open any YAML file in a text editor to see the current state, or even edit it manually if something goes wrong. There is no opaque database to query, no binary format to decode, no migration to run (beyond the one-time v1-to-v2 migration in ADR 019). The file-based approach aligns with the Unix philosophy of storing configuration and data in human-readable text files.

The TUI's live-update mechanism demonstrates how this works in practice. The TUI uses `fs.watch` with `{ recursive: true }` to monitor the entire `.pm/` directory tree. When any YAML file changes — whether from a CLI command, an MCP tool call, or a manual edit — the watcher fires, and the TUI re-reads the affected files and re-renders. This provides a live dashboard without any pub/sub system, WebSocket connection, or shared state.

The primary risk is concurrent writes. If two processes try to update the same YAML file simultaneously, one write could overwrite the other. In practice, this has not been a problem because pm is a single-user tool and AI agents operate sequentially (they await each tool call before making the next). But the architecture provides no protection against this scenario — there are no file locks, no optimistic concurrency checks, no compare-and-swap. If pm were ever used in a multi-user or multi-agent context, this would become a real problem that would require either file locking or a move to a coordination layer.

The lack of a service layer also means there is no centralized place to enforce authorization, rate limiting, or auditing. Every interface has full read-write access to all YAML files. For a personal productivity tool, this is appropriate. For a team tool, it would be a significant limitation.

Finally, the TUI's use of `fs.watch` with `{ recursive: true }` has a platform compatibility caveat. Recursive file watching is well-supported on macOS and Linux (via inotify), but behavior varies across operating systems and filesystem types. The current implementation handles this gracefully — if the watcher fails to initialize, it fails silently and the TUI still works, just without live updates. But this is a known fragility that would need revisiting if pm targeted platforms beyond macOS and Linux.
