# ADR 030: Epics as the Sole Organizational Boundary

## Status

Accepted

## Context

Project management tools typically offer multiple levels of temporal and thematic organization: sprints, milestones, releases, iterations, cycles, projects within projects. Each layer adds configuration, ceremonies (sprint planning, retrospectives), and state management (sprint capacity, velocity tracking, burndown charts). These constructs exist to coordinate human teams around shared deadlines and resource constraints.

Agent-pm serves a fundamentally different user: AI agents operating autonomously within a codebase, supervised by a human who checks status asynchronously. Agents do not have sprint ceremonies, do not negotiate scope at planning meetings, and do not need velocity-based forecasting. They execute stories in dependency order as fast as they can, filing reports and comments along the way. The human supervisor cares about what is done, what is in progress, what is blocked, and what needs attention -- not which sprint a story falls into.

Introducing sprint or milestone constructs would require temporal state management (start dates, end dates, capacity), status transitions (sprint active, sprint closed), and cross-cutting queries (show me all unfinished stories from last sprint). Each of these adds schema complexity, CLI commands, TUI views, and cognitive overhead without providing value to the agent-first workflow.

## Decision

We will use epics as the sole organizational boundary in the data model. An epic is a thematic grouping of related stories with a status (`backlog`, `in_progress`, `done`, `cancelled`), a title, a description, and an embedded array of stories (ADR-003). There are no sprints, milestones, iterations, releases, or any other temporal grouping construct.

Prioritization is handled through story ordering within an epic and per-story priority fields (`high`, `medium`, `low`), not through sprint assignment. Dependencies between stories are expressed via `depends_on` arrays referencing story codes, enabling dependency-aware execution ordering without temporal batching.

Epic status is derived from story statuses via `pm epic sync`: an epic with all stories done or cancelled is marked done; an epic with at least one in-progress story is marked in-progress; otherwise it remains in backlog. This makes epic status a computed aggregate rather than a manually managed state.

The `pm status` command provides the supervisory overview that sprints would normally serve: active epics, in-progress stories, completion percentages, and the next recommended story based on priority and dependency readiness. The TUI dashboard (ADR-013) renders this information in real time via file watching.

## Consequences

The data model is remarkably simple. Two entity types -- epics and stories -- with a single containment relationship. The hierarchical code scheme (ADR-005) supports up to 999 epics with 999 stories each, which is more than sufficient for the expected scale. Schema changes, validation logic, and CLI commands are all proportionally simpler than they would be with additional organizational layers.

Agents never need to answer "which sprint does this belong to?" or "is this sprint full?" They simply pick the highest-priority unblocked story and execute it. The dependency-aware orchestrator (`pm-work-on-project` slash command) automates this selection, iterating through stories in dependency order without any temporal batching concept.

The absence of milestones means there is no built-in mechanism for release planning or deadline tracking. Teams that need to ship a set of features by a date must manage that externally (in a spreadsheet, calendar, or a different tool). This is an intentional non-goal: agent-pm optimizes for autonomous execution throughput, not human ceremony coordination.

There is no velocity tracking, burndown chart, or capacity planning. Story points exist (Fibonacci values: 1, 2, 3, 5, 8) for relative sizing, but they are not aggregated into velocity metrics. The assumption is that agent execution speed is determined by model capability and task complexity, not by team capacity -- rendering velocity calculations meaningless.

If a team later needs sprint-like batching, they can approximate it by creating an epic per "sprint" and moving stories between epics. This is a workaround, not a first-class feature, and it would not provide sprint ceremonies or velocity tracking. The decision to avoid sprints is easy to reverse by adding a new entity type, but it would require changes across the schema, CLI, TUI, and MCP layers.
