# ADR 028: Structured Execution Reports for Agent Accountability

## Status

Accepted

## Context

When an agent completes a story, the code change is visible in the repository, but the reasoning behind the change is not. Why did the agent choose approach A over approach B? What assumptions did it make about the system? What did it intentionally defer? What might conflict with other agents' work? Without structured records of these decisions, the consolidation pipeline (ADR-026) has no raw material to synthesize, and humans reviewing agent work must reverse-engineer intent from diffs.

The execution report contract needed to capture five distinct categories of information: decisions made during implementation, assumptions about the system that informed those decisions, tradeoffs where alternatives were explicitly rejected, items deliberately placed out of scope, and potential conflicts with other tasks. Each category serves a different downstream consumer: decisions feed ADR synthesis, assumptions feed conflict detection, tradeoffs provide historical context, out-of-scope items inform future story creation, and potential conflicts trigger resolution tasks.

The report format also needed to distinguish between episodic observations (narrative context specific to this task's execution) and semantic observations (generalizable insights that are candidates for architectural knowledge). This distinction determines whether an observation can be promoted to an ADR during consolidation.

## Decision

We will require agents to file a structured execution report after completing each story, stored as `{story-code}-report.yaml` in `.pm/reports/`. The report schema is defined in `src/schemas/agent-report.schema.ts` and validated by Zod on every read and write.

Each report contains five array fields -- `decisions`, `assumptions`, `tradeoffs`, `out_of_scope`, and `potential_conflicts` -- capped at 10 items each with a 500-character limit per item. Every item in `decisions` and `assumptions` carries a `type` field: `episodic` for task-specific narrative or `semantic` for generalizable insight. The default is `episodic`, requiring agents to explicitly opt in to semantic classification when they believe an observation has broader relevance.

Tradeoffs are structured as `{alternative, reason}` pairs, capturing what was rejected and why. Potential conflicts carry a `confidence` level (`low`, `medium`, `high`) reflecting the agent's certainty that the conflict is real.

Reports carry a `consolidated` flag (initially `false`) that the consolidation pipeline sets to `true` after ingestion. A `status` field (`complete` or `partial`) allows agents to file interim reports when a story is not fully finished.

The CLI provides `pm report create` for filing and `pm report view` for reading. The MCP server exposes `pm_report_create` and `pm_report_view` tools. CLI input uses lightweight delimited formats: colon-prefixed type for decisions (`semantic:some text`), pipe-separated fields for tradeoffs (`alternative text | reason text`), and confidence-tagged strings for conflicts (`assumption | high | note`).

## Consequences

The consolidation pipeline has a well-defined input contract. Agent reports are the primary input to the structural dedup and LLM synthesis phases. The `type` field on decisions and assumptions directly influences whether an observation can be promoted to an ADR: only `semantic`-typed items are candidates for architectural knowledge promotion.

The 10-item cap and 500-character limit force agents to be selective about what they report. This is intentional: a report that captures everything is as useless as one that captures nothing. The constraints push agents toward the most significant observations. The trade-off is that genuinely complex stories may require the agent to compress or omit information.

Reports exist as one file per story, not per execution attempt. The `--force` flag allows overwriting a previous report if the story is re-executed. This means the history of failed attempts is not preserved in the report system (though it remains in git history if `.pm/` is tracked).

The episodic/semantic distinction is a judgment call made by the agent at filing time. Agents may miscategorize observations -- tagging routine decisions as semantic or failing to tag genuine insights. The consolidation pipeline's structural dedup and LLM synthesis provide a second pass that can catch miscategorizations, but the initial signal quality depends on agent behavior.

The delimited CLI input format is convenient for programmatic generation but fragile for human use. Pipes and colons in the content text must be avoided or escaped. This is acceptable because the primary consumers are agents using the MCP tools, where the structured parameters avoid delimiter issues entirely.
