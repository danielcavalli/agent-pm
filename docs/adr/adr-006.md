# ADR 006: CLI as the Core Execution Layer

## Status

Accepted

## Context

pm has multiple consumers that need to create and query project management data: human users typing commands in a terminal, an MCP server responding to AI agent tool calls, slash commands that orchestrate multi-step workflows, and an interactive TUI dashboard. Each of these interfaces needs access to the same validation logic, error handling, and data manipulation routines.

The most natural approach in a Node.js project would be to extract shared logic into a library layer — a set of exported functions that each interface imports and calls directly. This is what most frameworks encourage: the CLI becomes a thin wrapper around library calls, and the MCP server imports those same functions. This approach minimizes overhead and allows interfaces to share in-process state, connection pools, or caches.

However, a library-based approach creates tight coupling between the interfaces and the implementation. The MCP server would need to import TypeScript modules from the pm package, which means it must run in the same Node.js process or at least share the same module graph. Any change to function signatures, error types, or return values would ripple across all consumers simultaneously. Testing each interface in isolation becomes harder because they all depend on the same internal API surface.

An alternative is to treat the CLI as the canonical execution layer — the single place where all validation, data access, and business logic is exercised — and have every other interface delegate to it by invoking the `pm` binary as a subprocess. This means the MCP server does not import any pm library code; it constructs CLI argument arrays and calls `spawnSync("pm", args)`. Slash commands, being markdown prompt files interpreted by an AI agent, already invoke `pm` as a shell command naturally.

The tension is between performance and simplicity on one side (direct library calls are faster and share types) versus clean separation and operational independence on the other (subprocess delegation means each interface only depends on the CLI's public contract: its command-line arguments, exit codes, and stdout/stderr output).

## Decision

We designate the `pm` CLI as the single execution layer for all business logic. The MCP server constructs argument arrays and invokes `spawnSync("pm", cliArgs)` to fulfill tool calls. Slash commands instruct the AI agent to run `pm` subcommands.

The TUI is a partial exception to this rule. It reads YAML files directly for display via `src/lib/` imports (e.g., `fs.ts`, `codes.ts`, `agent-state.ts`) and performs limited writes for agent escalation responses (`writeAgentResponse()`). These writes bypass the CLI because the TUI's interactive response flow requires in-process state management that subprocess delegation cannot provide. All other mutations (story updates, epic adds, etc.) still go through the CLI.

The MCP server is the strictest adherent to this boundary: it has zero imports from `src/lib/`, `src/commands/`, or `src/schemas/`, depending only on `@modelcontextprotocol/sdk` and `node:child_process`.

## Consequences

The primary benefit is a clean architectural boundary. The MCP server has zero TypeScript dependencies on pm's internal modules — its entire source file (`mcp-server.ts`) imports only from `@modelcontextprotocol/sdk` and `node:child_process`. This means the MCP server and the CLI can be tested, built, and reasoned about independently. A refactor of pm's internal module structure (renaming files, changing function signatures, restructuring the lib directory) cannot break the MCP server as long as the CLI's command-line interface remains stable.

This design also simplifies the MCP server considerably. Rather than reimplementing argument parsing, validation, and error handling for tool calls, the MCP server is a thin translation layer: it maps MCP tool parameters to CLI flags and returns stdout as the tool result. The `runPm` helper function is roughly fifteen lines of code.

The subprocess approach introduces latency. Every MCP tool call spawns a new Node.js process, which includes the cost of process creation, module loading, and YAML parsing. For an interactive application with rapid-fire requests, this overhead would be unacceptable. For pm's use case — an AI agent occasionally filing stories or checking status during a coding session — the latency (typically 100-300ms per invocation) is negligible. The agent is already spending seconds generating responses, so a few hundred milliseconds for a tool call is imperceptible.

A subtler consequence is that the CLI's text output becomes a de facto API. The MCP server returns the CLI's stdout verbatim to the AI agent. This means changes to output formatting (column widths, emoji, color codes) can affect how agents parse and interpret results. The `--json` flag on the status command partially mitigates this by providing a structured output format, but most commands still rely on human-readable text. Extending `--json` support to other commands would strengthen this contract, though it has not been necessary so far because AI agents handle free-text output well.

There is also a deployment coupling: the MCP server assumes that `pm` is available on the system PATH. If the CLI is not installed globally (or `PM_HOME` is misconfigured), the MCP server will fail with an unhelpful "pm command failed" error. The installer (`install.sh`) mitigates this by performing `npm install -g .` before configuring the MCP server, but this remains a potential source of confusion if someone sets up the MCP server manually without installing the CLI first.
