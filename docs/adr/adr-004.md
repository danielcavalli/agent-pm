# ADR 4: Auto-Rebuilt Index File

## Context

The `pm status` command needs to show a summary of the current project -- name, status, epic count, story count, and completion progress. Loading this information requires reading `project.yaml` and every epic file in the local `.pm/` directory. For a small number of epics this is fast, but the design should anticipate growth.

An index file at `.pm/index.yaml` could serve as a cache of aggregate data, avoiding the need to scan all epic files on every status query. The question is how to keep this index consistent with the underlying data.

Three approaches were considered:

- **Event-driven updates** using file watchers to detect changes and update the index in real time. This adds complexity (watcher lifecycle management, edge cases around rapid writes) and is fragile across different operating systems.

- **Database-backed index** using SQLite or similar. This provides transactional consistency but contradicts the YAML-files-as-storage decision (ADR 1) and requires a separate synchronization mechanism.

- **Rebuild on mutation** -- every CLI command that modifies data (init, epic add, story add, story update) calls `rebuildIndex()` after the mutation completes. The index is a derived artifact, always reconstructable from the source YAML files.

## Decision

We maintain an auto-rebuilt `index.yaml` file at `.pm/index.yaml` within the local project directory. The `.pm/` directory is resolved by `getPmDir()` using the priority chain: `PM_HOME` environment variable, then `<git-root>/.pm`, then `<cwd>/.pm`.

The index is a **flat single object** (not a multi-project array). Its schema (`IndexSchema` in `src/schemas/index.ts`) contains:

```yaml
code: PM
name: Project Management for AI Agents
status: active
epic_count: 56
story_count: 224
stories_done: 221
last_updated: '2026-03-14'
```

The `rebuildIndex()` function in `src/lib/index.ts` reads the local `.pm/project.yaml` and all epic files in `.pm/epics/`, computes aggregate counts (excluding cancelled stories), and writes a fresh `index.yaml`. It is called after every mutating CLI command.

## Status

Accepted. Updated 2026-03-13 to reflect local-first single-project architecture (see ADR 19).

## Consequences

The index is always consistent with the underlying data after any CLI-mediated mutation. There is no stale cache problem as long as changes go through the CLI. The `rebuildIndex()` function is compact and runs in milliseconds for the expected data scale.

The negative consequence is that changes made by directly editing YAML files (bypassing the CLI) will not be reflected in the index until the next CLI mutation triggers a rebuild. This is acceptable because the index is a convenience for the summary view -- `pm status` can also read live data directly. The PRD's design principle that "files are the API" means direct edits are a supported workflow, and the index gracefully catches up on the next mutation.

Because pm now operates in single-project mode (one `.pm/` directory per repository), there is no need for incremental or per-project rebuild logic. The `rebuildIndex()` function always rebuilds the single index from scratch. The `_filterCode` parameter in the function signature is retained for backward compatibility but is unused.

Another consequence is that the index file is a derived artifact and should not be hand-edited. It is overwritten completely on each rebuild. This is documented implicitly by the auto-maintenance behavior but could surprise a developer who tries to edit it manually.
