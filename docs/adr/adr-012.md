# ADR 012: commander.js for CLI Framework

## Status

Accepted

## Context

pm needs a CLI framework to parse commands, subcommands, options, and arguments, generate help text, and dispatch to handler functions. The CLI surface is moderately complex: it has top-level commands (`init`, `status`, `work`, `migrate`, `tui`), nested subcommands (`epic add`, `epic list`, `epic sync`, `story add`, `story list`, `story update`), required and optional flags, variadic options (`--criteria` and `--tech-stack`), and a global `--debug` flag. The framework should handle all of this without requiring extensive boilerplate.

The three main contenders in the Node.js CLI framework space are commander.js, yargs, and oclif. Each reflects a different philosophy.

commander.js is a lightweight, fluent API for defining commands and options. It uses method chaining (`.command()`, `.option()`, `.action()`) to build a command tree, generates help text automatically, and stays out of the way. It has no opinions about project structure, plugin systems, or testing patterns. It is the most-used CLI framework in the npm ecosystem, which means good documentation and community support.

yargs takes a more configuration-heavy approach, with a builder pattern that can become verbose for nested subcommands. It offers powerful features like middleware, command modules, and automatic type coercion, but these features add API surface area that pm does not need. yargs also has a history of breaking changes between major versions, which has been a maintenance concern for long-lived projects.

oclif (from Salesforce/Heroku) is a full framework that imposes structure: commands are classes with decorators, plugins are loaded dynamically, and the project must follow a specific directory layout. It is designed for large, multi-plugin CLI suites like the Heroku CLI or the Salesforce CLI. For a focused tool like pm, oclif's structure would be overhead — more files per command, mandatory TypeScript decorators, a plugin manifest system — without proportional benefit.

## Decision

We use commander.js as the CLI framework. Commands and subcommands are defined in a single file (`src/cli.ts`) using commander's fluent API. Each command's action handler is wrapped in a lazy `import()` to defer loading of command modules until they are actually invoked. An `action()` wrapper function provides consistent error handling across all commands.

The CLI structure in `cli.ts` looks like this: top-level `program` defines the program name, description, and version, then each command is defined inline with its options and a lazy-loading action handler. Subcommand groups (`epic`, `story`) are created as intermediate `Command` instances, with their subcommands chained beneath them.

## Consequences

commander.js provides a clean, readable CLI definition. The entire command tree — all commands, subcommands, options, descriptions, and handler bindings — fits in a single 250-line file. A new contributor can read `cli.ts` top to bottom and understand every command pm supports, what arguments each takes, and where the implementation lives. This is a significant readability benefit over frameworks like oclif where command definitions are scattered across many files.

The lazy `import()` pattern in action handlers (`const { epicAdd } = await import("./commands/epic.js")`) means that running `pm status` does not load the code for `epic`, `story`, `init`, `migrate`, or `tui` commands. This keeps startup time low, especially for frequently-used commands. In a framework like oclif, this kind of lazy loading is built in (commands are discovered from the filesystem), but in commander.js it requires explicit use of dynamic imports. The tradeoff is that each action handler has a small amount of boilerplate — the `await import(...)` line — but this is consistent and easy to copy when adding new commands.

The `action()` wrapper function is a pm-specific pattern that works well with commander.js's callback-based action handlers. It converts async action functions into synchronous callbacks that commander expects, while adding centralized error handling: `PmError` instances are formatted with their error code, unexpected errors are shown with or without stack traces depending on the `--debug` flag. This pattern would work with any CLI framework, but commander.js's simplicity makes it easy to implement without fighting the framework.

commander.js does not provide type safety for parsed options. The `options` parameter in action callbacks is typed as `Record<string, unknown>`, and each command implementation must cast or validate the values it receives. This is a minor inconvenience — the commands immediately pass options through Zod validation (see ADR 011), so invalid values are caught early — but it means there is a brief window between commander's parsing and Zod's validation where option types are unverified. A more type-safe framework (or a TypeScript wrapper around commander) could eliminate this gap, but the practical risk is low given the immediate Zod validation.

commander.js has no built-in support for shell completions, configuration files, or plugin discovery. pm does not need any of these features today. If shell completions become desirable, there are community packages that generate them from commander command trees. If a plugin system were ever needed, it would be a significant departure from pm's current single-binary architecture and would likely warrant reconsidering the framework choice entirely.
