# ADR 023: Autonomous Swarm Self-Improvement

## Status

Proposed

## Context

### The Optimization Gap

agent-pm v0.1.0-alpha (ADR-021) introduces agent visibility through an observer model: agents write heartbeat files, the TUI displays fleet status, and humans handle escalations. The orchestrator (`/pm-work-on-project`) dispatches stories to parallel agents using a static strategy -- fixed concurrency, hardcoded sequential-within-epic and parallel-across-epic policy, no feedback from outcomes to future decisions.

But orchestration strategy is not one-size-fits-all. A greenfield sprint benefits from aggressive parallelism; a refactoring project needs sequential caution. A team with experienced agents can tolerate higher autonomous confidence thresholds; a new project needs tighter human oversight. Today, the human must manually tune these parameters through trial and error with no systematic feedback.

Two open-source autonomous research systems demonstrate that a prompt-driven experiment loop can systematically improve strategy through real-world observation:

**karpathy/autoresearch** proves that an AI agent can run an infinite experiment loop where it mutates a single configuration, evaluates against a fixed metric, and keeps or discards changes -- all orchestrated by a Markdown prompt file with zero framework code. The evaluation function (`prepare.py:evaluate_bpb()`) is immutable; the agent cannot game the metric. Negative results (crashes, regressions) are recorded alongside successes in `results.tsv`, preventing duplicate exploration.

**mutable-state-inc/autoresearch-at-home** adds a coordination layer for multiple experiment agents: a namespaced key-value store for shared state, an optimistic locking claim protocol with TTL to prevent duplicate work, cascading best-tracking with anomaly rejection, hypothesis exchange for cross-agent idea sharing, insight accumulation for collective learning, and an `analyze_swarm()` function for situational awareness. The entire coordination layer degrades gracefully -- if shared state is unavailable, agents continue solo.

### The Core Reframing: Task Board as Plan

In autoresearch, the mutable experiment file is ML model code (`train.py`) and the fixed metric is `val_bpb`. In agent-pm, the mutable search space has two layers:

**Layer A -- The task board itself.** agent-pm epics and stories ARE the plan. The `pm` CLI commands ARE the mutation operators. Most "strategy parameters" from a naive flat-config approach are already expressible through the task board:

- **Assignment policy** is expressed through story `priority` fields and `depends_on` relationships. `pm prioritize` already changes ordering. An experiment agent that wants to test "priority-weighted assignment" can reprioritize stories using `pm story update <CODE> --priority high`.
- **Same-epic sequencing** is expressed through `depends_on` within an epic. If S002 depends on S001, they are sequential. Removing the dependency allows parallel execution. The experiment agent adds or removes dependencies using `pm story update <CODE> --depends-on`.
- **Story granularity** affects throughput and quality. An experiment agent that splits an 8-point story into two 3-pointers is making a structural change no flat config file can express.

**Layer B -- Residual runtime config.** A small `strategy.yaml` contains parameters that genuinely cannot be expressed through the task board: concurrency limits, heartbeat timing, escalation thresholds, and observation window size.

This dual-layer model aligns with AFlow (Zhang et al., ICLR 2025 Oral), which found that treating workflow optimization as search over structured representations (not flat parameter vectors) enables smaller models to outperform GPT-4o at 4.55% of inference cost. The task board IS a structured workflow representation.

**The fixed metric** is derived from real project outcomes measured by per-project evaluation tactics (`tactics.yaml`). There is no synthetic benchmark. The swarm improves by experimenting on the actual project it manages.

### Relationship to v0.1.0-alpha

The swarm self-improvement layer depends on and extends the v0.1.0-alpha infrastructure:

- **E051 Agent State Layer** provides heartbeat data that feeds the `idle_ratio` tactic measurement
- **E052 MCP Tool Extensions** provides the agent-facing API pattern for new swarm tools
- **E053-E054 TUI** provides the display surface for experiment status
- **E055 Escalation Lifecycle** provides the mechanism for agent-suggested tactic changes and the escalation metric
- **E056 Technical Debt** provides advisory locking used by the claim protocol

The self-improvement layer is purely additive. Projects that do not initialize `.pm/swarm/` see no changes. Existing heartbeat, escalation, and story execution flows are unaffected.

### Requirements Alignment

This work addresses the optimization gap identified after the v0.1.0-alpha PRD was finalized. It does not duplicate or conflict with any v0.1.0-alpha requirements (M1-M7, S1-S8, C1-C4). It explicitly avoids Won't Have items (W1-W8). ADR-022 (dropping the pm-reminder plugin) is unrelated to this work.

## Decisions

### 1. Per-Project Evaluation Tactics with Tchebycheff Scalarization

Each project defines what "good orchestration" means through evaluation tactics in `.pm/swarm/tactics.yaml`. A tactic is a named metric with a measurement method, direction, and weight.

**Schema:**

```yaml
# .pm/swarm/tactics.yaml
version: 1
tactics:
  - name: throughput                    # unique identifier
    description: "Stories completed per wall-clock hour"
    metric: stories_per_hour            # programmatic key
    direction: higher_is_better         # higher_is_better | lower_is_better
    weight: 0.25                        # relative weight (all weights sum to 1.0)
    measurement: derived                # derived (computed from observation data)
    source: story_result                # story_result | heartbeat | escalation

profiles:
  balanced: { throughput: 0.25, first_pass_quality: 0.25, waste: 0.15, idle_time: 0.10, coordination_overhead: 0.10, escalation_responsiveness: 0.10, escalation_rate: 0.05 }
  velocity: { throughput: 0.40, first_pass_quality: 0.15, waste: 0.10, idle_time: 0.20, coordination_overhead: 0.05, escalation_responsiveness: 0.05, escalation_rate: 0.05 }
  quality:  { throughput: 0.10, first_pass_quality: 0.40, waste: 0.20, idle_time: 0.05, coordination_overhead: 0.15, escalation_responsiveness: 0.05, escalation_rate: 0.05 }
```

**Composite scoring via Tchebycheff scalarization.** The composite score is NOT a weighted sum. It uses Tchebycheff scalarization:

```
composite = min_i(w_i * normalized_metric_i)   # then maximize this
```

Weighted-sum scalarization can only find solutions on the convex hull of the Pareto front -- it provably misses "balanced" configurations where multiple metrics are moderately good (Lin et al., "Smooth Tchebycheff Scalarization for Multi-Objective Optimization", ICML 2024). Tchebycheff is Pareto-complete: it can find ALL optimal trade-offs regardless of Pareto front convexity. The tactics.yaml format, profiles, and weights are unchanged -- only the aggregation formula differs.

**Normalization via running statistics.** Metrics are normalized using incrementally updated mean and standard deviation (z-score normalization), not fixed min/max bounds. Naive min/max normalization degrades optimization when range estimates are inaccurate, which they will be during cold-start (He et al., "A Survey of Normalization Methods in Multiobjective Evolutionary Algorithms", IEEE TEC 2021). Running statistics are updated after each experiment, converging to accurate estimates within 5-10 experiments.

**Defaults and customization:**
- `pm swarm init` copies a default template (see `docs/templates/swarm-default-tactics.yaml`) into `.pm/swarm/tactics.yaml`
- The user edits tactics to match project priorities
- Agents can suggest tactic changes by filing a hypothesis with `type: tactic_suggestion`. The human reviews via the TUI escalation mechanism (ADR-021 Decision 4) and approves or rejects

**Immutability guarantee:** The experiment agent treats `tactics.yaml` as read-only during an experiment run. Only humans (direct edit) or human-approved tactic suggestions can modify it. This is the analog to autoresearch's immutable `prepare.py:evaluate_bpb()`.

**Measurement sources:**
- `story_result`: Computed from persisted story outcome records in `.pm/swarm/observations/`. The `/pm-work-on-project` orchestrator persists each sub-agent's `STORY_RESULT` to disk after parsing it. This persistence is a prerequisite for the evaluation engine.
- `heartbeat`: Computed from `.pm/agents/*.yaml` heartbeat timestamps (idle gaps, active durations) -- depends on E051
- `escalation`: Computed from escalation request/response data (count, latency, type distribution) -- depends on E055

**Rationale:** Per-project tactics avoid the "one metric to rule them all" trap. Tchebycheff scalarization ensures the optimizer finds balanced configurations, not just solutions that max one metric at the expense of others. Running statistics handle cold-start gracefully.

### 2. Dual-Layer Mutable Search Space

The experiment agent operates over two mutation layers, each with distinct mechanisms for applying and reverting changes.

**Layer A: Task board mutations (structural).**

The experiment agent modifies the task board using existing `pm` commands:
- `pm prioritize` -- reorder stories and epics
- `pm story update --priority` -- change individual story priority
- `pm story update --depends-on` -- add/remove dependency edges
- `pm story add` / `pm story update` -- split stories, adjust scope
- `pm epic add` -- create new organizational groupings

Board mutations are tracked through git (`.pm/` is already git-tracked). Each structural change is committed with a descriptive message. Keep/discard uses `git revert` on the board-change commit -- exactly autoresearch's pattern where each experiment is a commit and failures reset.

**Layer B: Runtime config (operational).**

A small `strategy.yaml` contains parameters that cannot be expressed through the task board:

```yaml
# .pm/swarm/strategy.yaml -- ONLY runtime parameters
version: 1
config_version: 1                         # monotonic counter for fencing (Decision 5)
parameters:
  dispatch:
    max_concurrent_agents: 5              # Task tool concurrency limit [1, 20]
  heartbeat:
    frequency_seconds: 15                 # how often agents report state [5, 60]
    stale_threshold_seconds: 60           # when to consider an agent gone [30, 300]
  escalation:
    confidence_autonomous: 0.85           # above this: agent decides alone [0.50, 1.00]
    confidence_review: 0.50               # below this: agent blocks and escalates [0.00, 0.85]
    max_pending_escalations: 3            # throttle to prevent alert fatigue [1, 10]
  experiment:
    observation_window_stories: 10        # stories to observe before evaluating [5, 50]
    claim_ttl_seconds: 900                # experiment claim expiry [300, 3600]
```

Notice what is NOT in strategy.yaml compared to early designs: `assignment_policy`, `same_epic_policy`, `similarity_threshold`, `min_experiments_before_revert`. These are either expressed through the task board (Layer A) or are internal experiment-agent logic (not orchestrator config).

**Relationship to `/pm-work-on-project`:** The orchestrator reads only `max_concurrent_agents` from `strategy.yaml` to limit concurrent dispatches. It already reads priorities and dependencies from the task board. If `strategy.yaml` is absent, it falls back to current hardcoded behavior.

**Strategy version tagging:** Each persisted observation record includes two attribution hashes:
- `strategy_hash`: SHA-256 of `strategy.yaml` at dispatch time
- `board_hash`: SHA-256 of a deterministic serialization of `.pm/index.yaml` + `.pm/epics/*.yaml` at dispatch time

The evaluation engine filters observations by these hashes, including only stories that ran under the experiment's configuration. This solves the attribution problem for both runtime and structural changes.

**Baseline establishment:** On first invocation (`.pm/swarm/best/` empty), `/pm-swarm-improve` runs a baseline observation under the current configuration, computes the initial composite score, and writes it to `.pm/swarm/best/metadata.yaml` before entering the experiment loop. Analogous to autoresearch's "run baseline to establish reference val_bpb."

**Parameter space boundaries:** Each numeric parameter has min/max bounds (shown in brackets above). The experiment loop can only propose values within these ranges. This prevents degenerate configurations.

**Rationale:** The dual-layer model is strictly more expressive than a flat config file. Structural mutations (reprioritizing, splitting stories, changing dependencies) capture optimization dimensions that no parameter file can represent. The residual config captures what is genuinely runtime. This aligns with AFlow's finding that structured workflow search outperforms flat parameter search.

### 3. File-Based SwarmStore

All inter-agent coordination state lives under `.pm/swarm/` as YAML files, consistent with the observer model established in ADR-021.

**Interface definition (`SwarmStore`):**

```typescript
interface SwarmStore {
  // Core CRUD
  read(namespace: string, key: string): Promise<Record<string, unknown> | null>;
  write(namespace: string, key: string, data: Record<string, unknown>): Promise<void>;
  list(namespace: string): Promise<string[]>;
  delete(namespace: string, key: string): Promise<void>;

  // Search
  search(namespace: string, query: string, threshold: number): Promise<Array<{ key: string; score: number }>>;

  // Claim protocol
  claim(namespace: string, key: string, agentId: string, ttlSeconds: number): Promise<boolean>;
  releaseClaim(namespace: string, key: string, agentId: string): Promise<void>;
  listActiveClaims(namespace: string): Promise<Array<{ key: string; agentId: string; expiresAt: string }>>;
}
```

**`FileSwarmStore` implementation:**

```
.pm/swarm/
  tactics.yaml              # evaluation tactic definitions (Decision 1)
  strategy.yaml             # residual runtime config (Decision 2, Layer B)
  observations/             # persisted STORY_RESULT records (written by /pm-work-on-project)
    <story-code>.yaml       # one file per story completion, includes strategy_hash + board_hash
  results/                  # experiment results (one file per experiment)
    <timestamp>-<agent-id>-<slug>.yaml
  claims/                   # active experiment claims
    <agent-id>-<slug>.yaml
  hypotheses/               # proposed experiments and tactic suggestions
    <timestamp>-<agent-id>-<slug>.yaml
  insights/                 # distilled observations
    <timestamp>-<agent-id>-<slug>.yaml
  best/                     # current best configuration + score
    strategy.yaml           # copy of the best-performing strategy.yaml
    board_commit: <sha>     # git commit hash of the best-performing board state
    metadata.yaml           # best score, experiment ref, timestamp, previous_best audit
```

**File naming:** `<ISO-timestamp>-<agent-id>-<kebab-slug>.yaml`. The timestamp prefix enables chronological listing via filesystem sort. The agent-id enables per-agent filtering. The slug provides human readability.

**Search implementation:** The `search()` method uses Levenshtein distance computed over YAML `description` fields. Levenshtein ratio is computed using a simple dynamic programming implementation (< 50 lines of TypeScript) with no external dependency. RETSim (Zhang et al., ICLR 2024) is a noted future alternative for lightweight near-duplicate text detection that outperforms character-based methods.

**Extension point for NetworkSwarmStore:** The `SwarmStore` interface is designed so that a `NetworkSwarmStore` implementation can be swapped in without changing coordination logic. The network variant would implement the same interface by calling an HTTPS API (similar to autoresearch-at-home's Ensue integration). This is explicitly out of scope but architecturally preserved.

**Graceful degradation:** If `.pm/swarm/` is missing or any file is corrupted, all `SwarmStore` operations return safe defaults (null, empty arrays, false for claims). Agents continue executing stories normally. The swarm layer is additive, never blocking.

**Rationale:** File-based storage follows the established pattern from `.pm/epics/`, `.pm/agents/`, `.pm/comments/`, and `.pm/reports/`. YAML files are human-readable, git-trackable, and require no infrastructure.

### 4. Experiment Loop as Slash Command with Dual Mutation Paths

The experiment loop runs as `/pm-swarm-improve`, a Markdown prompt file following the established slash command pattern. This is the analog to autoresearch's `program.md`.

**Loop structure:**

```
/pm-swarm-improve <PROJECT_CODE>

LOOP:
  1. ANALYZE  -- Read swarm state via pm swarm analyze. Output which mutation
     categories (board structure, runtime config) and specific dimensions
     (priorities, dependencies, concurrency, thresholds) have been explored
     in the last N experiments vs. which have not.

  2. HYPOTHESIZE -- Propose a specific change. Choose one of:
     - Board mutation: reprioritize stories, change dependencies, split a story
     - Runtime config: adjust a strategy.yaml parameter
     Prefer under-explored dimensions (rotation heuristic). For the first 5
     experiments, prioritize exploration; after that, focus on promising dimensions.

  3. CLAIM -- Write claim via pm swarm claim; verify ownership.

  4. CONFIGURE -- Branch by mutation type:
     If board mutation: execute the appropriate pm command(s), git commit.
       Record the git commit hash and current board_hash.
     If runtime config: write strategy.yaml with config_version+1, git commit.
       Record the strategy_hash.

  5. OBSERVE -- Watch next N stories complete under the new configuration.
     Read .pm/swarm/observations/ filtered by strategy_hash + board_hash.
     Read .pm/agents/ heartbeats for timing data.
     Read escalation files for escalation metrics.

  6. EVALUATE -- Compute each tactic metric. Compute Tchebycheff composite score.

  7. DECIDE -- Compare composite score against current best.
     If improved: KEEP. Update best.
     If not improved: DISCARD.
       For board mutations: git revert the board-change commit.
       For runtime config: restore strategy.yaml from best, increment config_version.

  8. PUBLISH -- Write result and insight (both mandatory). Optionally write
     follow-up hypothesis.

  9. GOTO 1
```

**Available pm commands as mutation operators.** The slash command prompt explicitly lists these as the experiment agent's toolkit for board mutations:
- `pm story update <CODE> --priority <high|medium|low>` -- change story priority
- `pm story update <CODE> --depends-on <CODE1,CODE2>` -- change dependency edges
- `pm story add --epic <EPIC> --title "..." --priority <P> --points <N>` -- add/split stories
- `pm prioritize <PROJECT>` -- bulk reorder
- `pm epic add --title "..." --priority <P>` -- organizational restructuring

**Observation, not execution:** The experiment agent does NOT dispatch or execute stories. It watches real outcomes produced by `/pm-work-on-project` running in parallel. It reads persisted observations, heartbeats, and escalation files. This separation is critical -- the experiment agent is a pure observer-optimizer.

**Human override handling (evaluated, not adopted blindly).** When the experiment agent detects an external change (strategy.yaml hash or board git history changed by someone other than the experiment agent):
1. The override takes effect immediately (human sovereignty preserved)
2. The experiment agent evaluates the human's configuration using the standard evaluation engine and records it as an experiment result
3. If the human's config scores better than the current best, it becomes the new best naturally through the normal keep/discard logic
4. If it scores worse, the agent has data to suggest reverting (via escalation, not automatic revert)

This preserves the no-harm property from CoExBO (Adachi et al., AISTATS 2024): even with suboptimal human interventions, the optimization can recover because it has comparative data.

**Structured output:** Each loop iteration emits a structured `EXPERIMENT_RESULT` block:

```yaml
EXPERIMENT_RESULT:
  experiment_id: "2026-03-14T10:30:00Z-agent-exp-01-raise-concurrency"
  mutation_type: runtime_config  # runtime_config | board_mutation
  hypothesis: "Increasing max_concurrent_agents from 5 to 7 will improve throughput"
  change_description: "dispatch.max_concurrent_agents: 5 -> 7"
  observation_window: 10 stories
  composite_score: 0.72
  previous_best_score: 0.68
  decision: keep  # keep | discard
  insight: "Higher concurrency improved throughput by 12% without increasing waste ratio"
```

**Rationale:** The prompt-as-orchestrator pattern matches autoresearch's `program.md` and agent-pm's existing slash commands. The dual mutation path lets the optimizer make structural changes (richer search space) alongside parameter tuning. The rotation heuristic prevents dimension starvation (inspired by AFlow's MCTS exploration). The evaluated-override model preserves human sovereignty while maintaining optimization convergence.

### 5. Claim Protocol with Fencing Tokens

Before starting an experiment, an agent writes a claim file to `.pm/swarm/claims/` to prevent duplicate work. This adapts autoresearch-at-home's optimistic locking protocol with fencing tokens for correctness.

**Protocol:**

```
1. CHECK EXACT: Read claims/. For parameter_change claims: match on parameter_path + new_value.
   For board_mutation claims: match on change_description. If active match exists, abort.
2. CHECK SIMILAR: For active claims of different type/target, compute Levenshtein
   similarity on descriptions. If any score exceeds 0.88, abort.
3. WRITE CLAIM: Write claim file with agent_id, type, description, claimed_at, ttl_seconds.
4. VERIFY: Wait 2 seconds. Re-read claim. If agent_id matches, claim is held.
5. FALLBACK: After 5 failed attempts, proceed anyway (solo mode).
```

**Claim file schema:**

```yaml
# .pm/swarm/claims/<agent-id>-<slug>.yaml
agent_id: "agent-exp-01"
type: runtime_config  # runtime_config | board_mutation | tactic_suggestion
description: "Increase max_concurrent_agents from 5 to 7"
parameter_path: "dispatch.max_concurrent_agents"  # for runtime_config
new_value: 7                                       # for runtime_config
pm_commands: []                                    # for board_mutation: list of pm commands to execute
claimed_at: "2026-03-14T10:30:00Z"
ttl_seconds: 900
status: active  # active | completed | expired
```

**Fencing tokens.** The write-wait-verify protocol is necessary but not sufficient. A paused agent can resume believing it still holds a claim after TTL expiry (Kleppmann, "How to Do Distributed Locking", 2016).

- **For runtime config:** `strategy.yaml` includes a monotonic `config_version` counter. The experiment agent reads the version before CONFIGURE, writes its change with `config_version + 1`, and verifies the version after writing. A version mismatch means another write occurred -- discard and re-analyze.
- **For board mutations:** Git commit hashes serve as natural fencing tokens. After committing a board change, the agent verifies its commit is still HEAD before proceeding to OBSERVE. If HEAD has moved (another agent or human committed), the agent's change may have been superseded.

**TTL enforcement:** Claims older than `ttl_seconds` from `claimed_at` are treated as expired at read time. No background cleanup needed.

**Why Levenshtein at 0.88 threshold:** Character-level similarity at 0.88 avoids false positives on opposite-direction changes (e.g., "Increase X from 5 to 7" vs "Decrease X from 5 to 3" have ~0.80 ratio, below threshold). The compound-key exact match (`parameter_path + new_value` or `change_description`) catches same-target duplicates; Levenshtein catches cross-target near-duplicates.

**Rationale:** The claim protocol prevents wasted computation from duplicate experiments. Fencing tokens handle the stale-claim problem that write-wait-verify alone cannot. The graceful fallback to solo mode ensures the protocol never blocks real work.

### 6. Inter-Agent Knowledge Sharing

Four knowledge artifacts enable collective learning across experiment agents.

**6a. Result Publishing**

Every experiment publishes a full result to `.pm/swarm/results/` regardless of outcome. Negative results are first-class data.

```yaml
# .pm/swarm/results/<timestamp>-<agent-id>-<slug>.yaml
experiment_id: "2026-03-14T10:30:00Z-agent-exp-01-raise-concurrency"
agent_id: "agent-exp-01"
mutation_type: runtime_config  # runtime_config | board_mutation
description: "Increase max_concurrent_agents from 5 to 7"
change_details:
  parameter_path: "dispatch.max_concurrent_agents"  # runtime_config
  old_value: 5
  new_value: 7
  # OR for board_mutation:
  # pm_commands: ["pm story update PM-E057-S003 --priority high"]
  # board_commit: "abc1234"
status: keep  # keep | discard | error
observation_window_stories: 10
tactic_scores:
  throughput: { value: 3.2, direction: higher_is_better }
  first_pass_quality: { value: 0.85, direction: higher_is_better }
  waste: { value: 0.12, direction: lower_is_better }
  idle_time: { value: 0.08, direction: lower_is_better }
  coordination_overhead: { value: 0.05, direction: lower_is_better }
composite_score: 0.72
delta_vs_previous_best: +0.04
strategy_snapshot: { ... }     # full strategy.yaml content
board_hash: "sha256..."        # board state hash
started_at: "2026-03-14T10:30:00Z"
completed_at: "2026-03-14T11:15:00Z"
```

**Cascading best-tracking** (adapted from autoresearch-at-home):
- **Agent personal best:** Updated in `.pm/swarm/best/agent-<id>.yaml`
- **Global best:** Updated in `.pm/swarm/best/strategy.yaml` and `metadata.yaml` with safety guards: reject `composite_score <= 0` (error), reject improvement > 0.30 in one step (anomaly), double-read before write
- **Previous best preserved:** `metadata.yaml` stores `previous_best_score` and `previous_best_experiment_id` for audit trail

**6b. Hypothesis Exchange**

Agents propose experiments for other agents to discover and execute.

```yaml
# .pm/swarm/hypotheses/<timestamp>-<agent-id>-<slug>.yaml
agent_id: "agent-exp-01"
title: "Reprioritize E057-S003 to high"
type: board_mutation  # parameter_change | board_mutation | tactic_suggestion
hypothesis: "The claim schema (S003) is blocking the evaluation engine. Raising its priority will cause the orchestrator to schedule it earlier, improving throughput."
suggested_change:
  pm_commands:
    - "pm story update PM-E057-S003 --priority high"
  expected_effect: "S003 scheduled in earlier tier, unblocking E059-S001"
evidence_keys:
  - "2026-03-14T09:00:00Z-agent-exp-02-sequential-same-epic"
priority: 2  # 1 (highest) to 5 (lowest)
status: unclaimed  # unclaimed | claimed | completed | rejected
created_at: "2026-03-14T10:30:00Z"
```

Three hypothesis types:
- `parameter_change`: proposes a runtime config change to strategy.yaml
- `board_mutation`: proposes a structural change via pm commands
- `tactic_suggestion`: proposes a new tactic or weight change (flows through escalation for human approval)

**6c. Insight Accumulation**

After every experiment, the agent posts a mandatory distilled observation about WHY something worked or failed.

```yaml
# .pm/swarm/insights/<timestamp>-<agent-id>-<slug>.yaml
agent_id: "agent-exp-01"
insight: "Higher concurrency (7 vs 5) improved throughput by 12% without increasing waste. Stories are largely independent across epics, so parallelism has room to grow."
evidence_keys:
  - "2026-03-14T10:30:00Z-agent-exp-01-raise-concurrency"
tags:
  - concurrency
  - throughput
posted_at: "2026-03-14T11:16:00Z"
```

Insights are mandatory per the PUBLISH contract: every loop iteration produces exactly one result and one insight. Hypotheses are optional (proposing a future idea is different from reflecting on what happened).

**6d. Swarm Analysis Aggregation**

`pm swarm analyze` aggregates all swarm state into a situational awareness snapshot:

```yaml
global_best:
  composite_score: 0.72
  experiment_id: "..."
  strategy: { ... }
recent_results: [...]
active_claims: [...]
unclaimed_hypotheses: [...]
agent_bests: [...]
improvement_trend: improving  # improving | plateaued | regressing
experiment_count: 7
exploration_coverage:          # which dimensions have been explored
  runtime_config: { max_concurrent_agents: 3, confidence_autonomous: 1, ... }
  board_mutations: { priority_changes: 2, dependency_changes: 0, story_splits: 0 }
```

**Trend detection:** Compare average composite score of 5 most recent experiments vs 5 preceding. Delta > 0.02: `improving`. Within +/- 0.02: `plateaued`. Delta < -0.02: `regressing`.

**Exploration coverage:** The analysis tracks which mutation categories and specific dimensions have been explored. This feeds the rotation heuristic in step 1 (ANALYZE), preventing dimension starvation.

**Rationale:** The four knowledge artifacts map directly from autoresearch-at-home's proven patterns. The exploration coverage addition ensures systematic search rather than repeated tweaking of the same dimension.

## Consequences

### Positive

- **Self-improving orchestration.** The swarm systematically discovers better configurations, replacing manual tuning with data-driven optimization.
- **Richer search space.** The dual-layer model lets the optimizer make structural changes (reprioritize, restructure dependencies, split stories) alongside parameter tuning -- dimensions no flat config file can express.
- **Per-project customization.** Tactics make "good" explicit and project-specific. Tchebycheff scalarization finds balanced configurations that weighted-sum misses.
- **Human sovereignty with no-harm guarantee.** Human overrides take effect immediately. The optimizer evaluates them as data points rather than adopting them blindly, preserving convergence even after suboptimal human interventions (CoExBO property).
- **Knowledge accumulation.** Negative results and mandatory insights prevent the swarm from repeating mistakes.
- **No new dependencies.** Levenshtein similarity, YAML files, Zod schemas, and existing pm CLI commands.
- **Backward compatible.** `.pm/swarm/` created only on explicit `pm swarm init`.
- **Graceful degradation.** Missing/corrupt `.pm/swarm/` -> agents continue normally.

### Negative

- **Observation latency.** Each experiment requires `observation_window_stories` (default 10) real stories. With 5 agents at ~2 stories/hour, one experiment takes ~1 hour.
- **Cold start.** First experiments have no prior data. Running-statistics normalization handles this gracefully but the first 5 composite scores will have high variance.
- **Board mutation reversibility.** `git revert` on board changes can produce merge conflicts if other changes landed between the experiment commit and the revert. The slash command must handle this (attempt revert, escalate to human if it fails).
- **File system accumulation.** Long-running swarms accumulate YAML in `.pm/swarm/results/` and `.pm/swarm/insights/`. Extending `pm gc` for swarm data is noted as future work.

### Extrinsic Metacognition Limitation

The experiment loop is a fixed structure (ANALYZE-HYPOTHESIZE-CLAIM-CONFIGURE-OBSERVE-EVALUATE-DECIDE-PUBLISH). The agent cannot change the loop itself -- by Liu & van der Schaar's taxonomy ("Truly Self-Improving Agents Require Intrinsic Metacognitive Learning", ICML 2025), this is extrinsic metacognition: self-improvement within an externally imposed framework.

The dual-layer model partially addresses this. By giving the experiment agent access to task board mutations, the agent can change the STRUCTURE of the work plan, not just runtime parameters. The `tactic_suggestion` hypothesis type further extends this by allowing agents to propose changes to the evaluation criteria themselves. Together these are steps toward intrinsic metacognition, though the loop structure remains fixed.

### What This Enables (future path)

- **NetworkSwarmStore** for multi-machine swarms (interface defined in Decision 3)
- **Embedding-based insight search** via an optional `EmbeddingSwarmStore` decorator (RETSim or similar)
- **Auto-calibrated confidence thresholds** using escalation outcome data
- **Cross-project knowledge transfer** by sharing insights between project swarms
- **GC for swarm data** extending `pm gc` to archive old results, expired claims, and stale insights
- **Dynamic weight adaptation** during optimization (Lu et al., 2025)
- **Loop structure modification** -- agents proposing changes to the experiment loop itself (e.g., "skip CLAIM for solo-agent mode", "add a VALIDATE step between CONFIGURE and OBSERVE")
- **MCTS-based exploration** replacing the rotation heuristic with structured tree search per AFlow
