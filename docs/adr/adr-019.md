# ADR 019: Bidirectional Migration Between Source, Global, and Local Data Directories

## Status

Accepted. Updated 2026-03-13 to document `pm migrate to-local` and the local-first architecture shift.

## Context

pm originally stored project data in a `projects/` directory within the pm repository itself. This was convenient during early development -- the data was versioned alongside the code, and there was no need to manage external file paths. But it had a fundamental limitation: the project data was coupled to the repository checkout. If you cloned the repo elsewhere, the data was not available. If you deleted the repo, you lost your project data. And the MCP server, which runs as a global service, had no reliable way to find the data directory because it did not know where the repo was checked out.

ADR 002 established `~/.pm/` as the global data directory, resolved via the `PM_HOME` environment variable (defaulting to `~/.pm`). This gave pm a stable, repository-independent location for project data. But existing users had data in the old `./projects/` layout that needed to be moved to the new `~/.pm/projects/` location.

With the shift to a local-first, single-project architecture (see ADR 4), pm now stores project data in a `.pm/` directory at the repository root. This means there is also a need to migrate in the reverse direction -- from the global `~/.pm/projects/{CODE}/` layout back to a local `.pm/` directory within a repository. Both migration directions are supported as explicit commands.

The migration could be handled in several ways. An automatic migration on first run would detect the old layout and silently move files, but this is risky -- moving files without explicit user consent can lead to confusion, and if something goes wrong, the user may not realize what happened. A flag-day approach (just switch the default and let users figure it out) would be simpler to implement but hostile to users who do not read changelogs. An explicit migration command gives the user control over when and how the migration happens, lets them verify the results, and keeps the process reversible.

## Decision

### `pm migrate from-source` (source to local)

This subcommand copies project data from a source location (defaulting to `./projects/`) into the local `.pm/` directory. It handles both single-project sources (a directory containing `project.yaml` directly) and multi-project sources (a directory containing `{CODE}/project.yaml` subdirectories, in which case only the first project is migrated since pm now operates in single-project mode).

The migration is a copy, not a move -- source files are left in place so the user can verify the result before manually cleaning up. If `.pm/project.yaml` already exists, the migration is skipped to prevent overwriting existing data. After copying, the command calls `rebuildIndex()` to ensure the local index is up to date.

The command accepts a `--source` flag to specify a non-default source directory, allowing migration from any location.

### `pm migrate to-local` (global to local)

This subcommand copies a project from the global `~/.pm/projects/{CODE}/` directory to a local `.pm/` directory at a specified target path. It flattens the directory structure by removing the `{CODE}/` nesting level.

Required flags:
- `--code <CODE>` -- the project code to migrate (e.g., `PM`)
- `--target <PATH>` -- the directory where `.pm/` will be created

Optional flags:
- `--cleanup` -- after a successful migration and verification, remove the source project from global storage (`~/.pm/projects/{CODE}/`)

The command copies `project.yaml`, `epics/`, `comments/`, `adrs/`, `reports/`, and `index.yaml` from the global project directory into `<target>/.pm/`. After copying, it runs a verification step that loads the migrated `project.yaml` and all epic files through their Zod schemas to confirm the data is valid. If verification fails, the command throws an error so the user knows the migration may be incomplete.

If the destination `.pm/` directory already exists, the command refuses to proceed and asks the user to remove it first or choose a different target. This prevents accidental overwrites.

## Consequences

The explicit migration commands respect the user's agency. Rather than silently relocating data, they require the user to invoke the appropriate subcommand, see the results, and decide when to clean up old files.

The copy-not-move semantics (in both directions) make migrations non-destructive by default. If a migration produces unexpected results, the original data is still intact at the source location. The user can fix the issue and re-run the migration, or manually copy individual files. This safety net is worth the small cost of temporary disk space duplication.

For `pm migrate from-source`, the skip-if-exists behavior prevents accidental data loss from repeated migration runs. If `.pm/project.yaml` already exists, the command prints a warning and exits without modifying anything.

For `pm migrate to-local`, the `--cleanup` flag provides an opt-in mechanism to remove the global copy after successful migration and verification. Without this flag, data exists in both locations until the user manually cleans up. The verification step (schema validation of migrated files) provides confidence that the copy is complete before any cleanup occurs.

The `rebuildIndex()` call at the end of `from-source` migration ensures the local index is consistent with the newly copied data. The `to-local` migration copies the existing `index.yaml` directly rather than rebuilding, since it preserves the data as-is.

One limitation is that the migration only copies files -- it does not transform them. If the data format changed between versions, a separate format migration would be needed. The `pm migrate` subcommands are specifically about relocating data between directory layouts; they are not a general-purpose data migration framework.

These migration commands support the transition from the earlier global `~/.pm/` architecture to the current local-first `.pm/` architecture. They remain in the codebase as they cost little to maintain and serve users who need to move data in either direction.
