# ADR 032: Retain Ink as the Primary TUI Framework

## Status

Accepted

## Context

The current `pm tui` implementation uses Ink and React (ADR-013). That choice fits the codebase's TypeScript and component-driven architecture, but it has known interaction gaps: native mouse click handling, built-in scroll containers, and tree/list primitives all require additional custom work.

Story `PM-E069-S002` evaluated whether `blessed` or `neo-blessed` should replace Ink as the foundation for the TUI. A spike prototype was built for the project tree panel using blessed, with direct evaluation of mouse support quality, rendering behavior, API ergonomics, maintenance risk, and community health.

The spike confirmed that blessed is stronger than Ink for low-level terminal widgets. Mouse wheel events, scrollbars, clickable list items, and boxed layouts are available out of the box. However, the prototype also showed that the programming model becomes substantially more imperative than the current Ink code: widget references are mutated directly, derived state must be synchronized manually, and the resulting structure is less aligned with the project's existing React-centric patterns.

Community signals also do not justify a framework migration. `blessed` remains widely used, but its ecosystem is older and largely stable rather than fast-moving. `neo-blessed` positions itself as a maintained fork, yet its observable adoption and package activity are lower than `blessed`.

## Decision

We retain Ink as the primary TUI framework for agent-pm.

Blessed remains acceptable for targeted prototypes and future spikes, but we do not migrate the production TUI to blessed or neo-blessed at this stage. Interaction gaps in the current TUI should continue to be addressed incrementally within the Ink architecture unless a future product requirement makes a click-heavy, widget-first terminal UI essential.

## Consequences

The existing `src/tui/` architecture remains intact. We preserve the current React component model, hooks, test strategy, and design-spec alignment instead of incurring a framework rewrite.

We accept that some interaction primitives will remain more manual in Ink. Features like mouse region mapping, custom scrolling, and specialized inputs may require additional implementation effort compared with blessed.

If future requirements shift toward dense mouse-driven workflows, a separate blessed entry point can be prototyped again without discarding the current Ink implementation prematurely. The spike from `PM-E069-S002` provides a runnable reference for that reevaluation.
