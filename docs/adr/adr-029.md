# ADR 029: TTL-Based Garbage Collection with Archive Strategy

## Status

Accepted

## Context

The commentary (ADR-027) and execution report (ADR-028) systems generate a steady stream of YAML files in `.pm/comments/` and `.pm/reports/`. The consolidation pipeline (ADR-026) ingests these artifacts and produces durable outputs (ADRs and resolution tasks), but the source artifacts remain on disk indefinitely unless explicitly cleaned up. Over the lifetime of a project, thousands of consolidated comments and archived reports accumulate, increasing directory scan times, index file sizes, and the cognitive load on developers browsing the `.pm/` directory.

The system needs a garbage collection mechanism that removes artifacts after they have served their purpose, while respecting two constraints: artifacts must not be deleted before the consolidation pipeline has processed them, and reports should be recoverable in case a consolidation run produced incorrect outputs.

Additionally, the YAML-based ADR system in `.pm/adrs/` (used by the consolidation agent, distinct from the Markdown ADRs in `docs/adr/`) accumulates entries that are superseded by newer decisions. These should be marked as superseded rather than deleted, preserving the historical record.

## Decision

We will implement TTL-based garbage collection in `src/commands/gc.ts` with per-artifact-type policies configured in `project.yaml` under the `gc_config` section.

**Comments** (default TTL: 30 days) are deleted outright. A comment is eligible for GC only when three conditions are met: (1) `consolidated` is `true`, meaning the consolidation pipeline has already ingested it; (2) the comment's age exceeds the TTL; and (3) either the comment's own author agent ID appears in `consumed_by` (applicable only to agent-authored comments) or the target task has reached `done` status. Human-authored comments can never satisfy the consumed check and are only eligible for GC when their target task completes. This triple condition ensures that unconsumed comments on active tasks are preserved even after consolidation, giving agents a chance to read them.

**Reports** (default TTL: 7 days) are archived, not deleted. Eligible reports (consolidated and older than TTL) are moved via `fs.renameSync` to `.pm/reports/archive/`. Archival rather than deletion preserves the ability to audit consolidation outputs against their source reports. The archive directory grows unboundedly but contains only small YAML files.

**ADRs** (default TTL: 90 days) in `.pm/adrs/` are marked as superseded by updating their status field in the ADR index (`ADR-000.yaml`). Supersession is detected through two paths: forward references (an accepted ADR's `references` array contains a `supersedes` entry) and backward references (an ADR has a `superseded_by.by_adr_id` field set). Superseded ADRs remain in the index for historical reference.

The GC command is `pm gc run` with `--dry-run` and `--verbose` options. All three artifact types are processed sequentially in a single invocation; per-type filtering is not currently supported. The MCP server exposes `pm_gc_run` for agent-triggered collection.

TTL values are configurable per-project in `project.yaml`'s `gc_config` section: `ttl_comments_days`, `ttl_reports_days`, `ttl_adrs_days`. Defaults are used when `gc_config` is absent.

## Consequences

The three-strategy approach (delete, archive, mark) reflects the different value propositions of each artifact type. Comments are ephemeral observations whose value is fully captured by the consolidation pipeline. Reports are the evidence chain for consolidation outputs and may need auditing. ADRs are historical records that should never be destroyed, only superseded.

The 7-day default TTL for reports is aggressive -- it assumes the consolidation pipeline runs frequently enough that reports are ingested within a week. Projects that run consolidation less frequently should increase this value. Comments at 30 days and ADRs at 90 days provide more buffer.

The `consolidated: true` prerequisite means GC cannot run ahead of consolidation. A project that never runs consolidation will accumulate artifacts indefinitely. This is by design: deleting un-ingested artifacts would lose information.

The consumed-or-done condition for comments means some comments will persist well beyond their TTL if the target task is never completed and no agent marks them as consumed. This is a conservative choice: it is better to keep a stale comment than to delete one that might still be relevant to an in-progress task.

The archive directory for reports grows without bound. There is no secondary GC for archived reports. If disk space becomes a concern, manual cleanup of the archive is required. For a project management tool generating kilobyte-scale YAML files, this is unlikely to be a practical issue.

The GC runs sequentially across artifact types (comments, then reports, then ADRs) within a single invocation. There is no parallelism, but the operations are fast (directory scans and file moves/deletes) and the total artifact count is bounded by project scale.

Missing or unparseable timestamps in artifacts are treated as infinitely old (the `ageDays()` helper returns `Infinity`), ensuring that malformed artifacts are always eligible for GC rather than accumulating permanently.
