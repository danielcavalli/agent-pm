# ADR 013: ink and React for Interactive TUI

## Status

Accepted

## Context

pm includes an interactive terminal dashboard (`pm tui`) that displays project status, epic progress, and story states in a live-updating view. The TUI needs to render structured layouts (tables, progress bars, headers), respond to keyboard input for navigation, and update automatically when the underlying YAML files change on disk. It is a read-only dashboard — users view data but do not mutate it through the TUI.

Building a terminal UI is a well-explored problem with several established approaches. The most primitive is writing raw ANSI escape codes to control cursor position, colors, and text formatting. This gives maximum control but requires managing the terminal state manually — clearing regions, tracking cursor position, handling resize events, and dealing with platform-specific terminal quirks. For anything beyond a simple progress bar, raw ANSI is tedious and error-prone.

Libraries like `blessed` and `terminal-kit` provide widget abstractions over ANSI codes — panels, lists, tables, input boxes — similar to a desktop GUI toolkit. They were popular in the early Node.js era for building complex terminal applications like dashboards and file managers. However, they follow an imperative programming model: you create widget instances, set properties, and call methods to update the screen. Managing state changes across multiple widgets requires manual coordination, and the resulting code tends to be difficult to reason about as complexity grows.

ink takes a fundamentally different approach: it brings React's component model to the terminal. You write JSX components that describe what the terminal should look like, React manages the component tree and reconciliation, and ink translates the virtual DOM into terminal output using Yoga (the CSS flexbox layout engine). State is managed with React hooks (`useState`, `useEffect`), and re-renders happen automatically when state changes — exactly as in a React web application.

The appeal of ink is that anyone who knows React already knows how to build with ink. The mental model is identical: components receive props, maintain state, and declare their UI as a function of that state. The terminal layout uses flexbox (via Yoga), so concepts like `flexDirection`, `padding`, and `gap` work as expected. Custom hooks can encapsulate reusable behavior — for example, a `useFileWatcher` hook that sets up `fs.watch` and triggers re-renders when files change.

## Decision

We use ink (v6) and React (v19) for the interactive TUI. The TUI is implemented as a set of React components in `src/tui/`, with a top-level `App` component rendered by ink's `render()` function. State management uses React's built-in hooks. File watching is implemented as a custom `useFileWatcher` hook that calls `fs.watch` with `{ recursive: true }` on the pm data directory and debounces change events by 300ms to avoid excessive re-renders.

The `tsconfig.json` includes `"jsx": "react"` to enable JSX compilation for `.tsx` files. React and ink are listed as production dependencies in `package.json` because the TUI ships as part of the `pm` binary.

## Consequences

The React component model makes the TUI code approachable for anyone familiar with modern React. A component that displays an epic's progress bar is a function that takes the epic as a prop and returns JSX — there is no imperative state management, no widget lifecycle to manage, and no manual screen clearing. When the file watcher detects a change, the top-level state updates, React re-renders the affected components, and ink efficiently updates only the changed portions of the terminal output.

The `useFileWatcher` hook encapsulates the file-watching complexity in a reusable abstraction. It sets up `fs.watch` with recursive monitoring, debounces rapid change events (multiple stories updated in quick succession during a batch operation would otherwise trigger dozens of re-renders), and cleans up the watcher when the component unmounts. This separation of concerns means the rendering components do not need to know about filesystem events — they just receive updated data as props.

However, ink and React together add a substantial dependency footprint. React alone is a large library, and ink pulls in Yoga (compiled from C++ to WebAssembly), a reconciler, and several supporting packages. For a TUI that is one of four interfaces and is used occasionally (not on every pm invocation), this is a significant cost. The lazy-loading pattern in `cli.ts` mitigates the startup impact — ink and React are only imported when `pm tui` is invoked — but they still contribute to the overall `node_modules` size and installation time.

The `jsx: "react"` option in `tsconfig.json` affects the entire compilation, not just TUI files. This means the JSX transform is available in any `.tsx` file, even if it is not in the `src/tui/` directory. In practice, only TUI components use JSX, but the compiler option is global. A more surgical approach would use a separate `tsconfig.tui.json` with JSX enabled, but the added build complexity is not justified for a project of pm's size.

The `fs.watch` approach for live updates has a known platform limitation. Recursive watching via `{ recursive: true }` relies on operating system support (inotify on Linux, FSEvents on macOS). The implementation handles this pragmatically: if recursive watching is not supported, the watcher fails silently and the TUI still works — it simply does not update live. Users on unsupported platforms would need to restart the TUI to see changes. This is documented in a code comment in `useFileWatcher.ts` but is not surfaced to the user at runtime, which could be confusing if someone expects live updates and does not see them.

One design tradeoff worth noting is that the TUI is read-only. It does not provide input forms for creating stories or updating statuses — those operations go through the CLI or MCP tools. This keeps the TUI simple (no form validation, no write-path error handling, no optimistic updates) but means it serves a narrower purpose than a full-featured terminal application. If write capabilities were added in the future, ink's form input components and React's state management would make this feasible without a rewrite, but it would significantly increase the TUI's complexity.
