# ADR 024: Local-First Single-Project Storage

## Status

Accepted

## Context

The original storage architecture (ADR-002) placed all project data under a global directory at `~/.pm/projects/`, keyed by project code. This made the `pm` binary a global orchestrator: any agent in any repository could reach any project's data through the global path. The `PM_HOME` environment variable allowed overriding the base directory for testing or CI.

In practice, the global approach introduced friction. Agents working inside a repository had no way to discover that a `.pm` project existed for that repo without being told the project code. The global directory also meant project data lived outside the repository's version control, making it impossible to track management artifacts alongside the code they described. Teams using `git` to collaborate on code could not use the same workflow to collaborate on project status, and CI pipelines had no access to project state without mounting the host's `~/.pm` directory.

An alternative was to place `.pm/` at the repository root, co-locating project data with the code it governs. This would make discovery automatic, enable git tracking of management artifacts, and remove the requirement for agents to know a project code before interacting with a project. However, it would also constrain each repository to a single project and require changes to path resolution, MCP semantics, the installer, and the TUI's file watcher scope.

## Decision

We will store all project data in a `.pm/` directory at the repository root, detected by finding the nearest `git` root via `git rev-parse --show-toplevel` and falling back to the current working directory. The `PM_HOME` environment variable continues to take precedence when set, supporting test harnesses and CI environments that need deterministic paths. The resolution order is: `PM_HOME` > git root > cwd.

We will enforce safety guards in `ensurePmDir()` (via its internal `determinePmLocation()` helper) that refuse to create `.pm/` directly inside the filesystem root (`/`) or the user's home directory (`$HOME`), preventing accidental data placement in sensitive locations. The read-only `getPmDir()` function does not enforce these guards -- it only locates an existing `.pm/` directory. When `PM_HOME` is set, the creation guards are bypassed, allowing test harnesses to use arbitrary paths.

The global `~/.pm/projects/` path is retained only as a migration source. The `pm migrate to-local` command (ADR-019) copies data from the global store into a repository's local `.pm/`, and `pm migrate from-source` handles arbitrary source directories.

Each repository hosts exactly one project. Multi-project scenarios are handled by having multiple repositories, each with its own `.pm/` directory.

## Consequences

Project data now travels with the repository. Cloning a repo that tracks `.pm/` gives the new developer immediate access to all epics, stories, and management artifacts. This is the single largest improvement to onboarding and collaboration.

The `workdir` parameter on every MCP tool becomes essential rather than optional. When an MCP server instance serves agents across multiple repositories, each tool call must specify the working directory so that `getPmDir()` resolves to the correct `.pm/`. Prior to this change, the global path made `workdir` unnecessary.

The one-project-per-repo constraint means monorepos with multiple logical projects cannot use `pm` without restructuring. This is an intentional simplification: the data model, CLI, and TUI all assume a single project context, and supporting multiple projects per directory would complicate every layer.

The `PM_HOME` override is now a testing affordance rather than the primary mechanism. Production usage defaults to git-root discovery. This reduces the surface area for misconfiguration but means agents running outside a git repository (rare but possible) must either set `PM_HOME` or accept cwd-based resolution.

Index rebuilds, file watches, and garbage collection all operate within the repository-local `.pm/` directory, which bounds their scope to a single project's data. This makes the system's I/O footprint proportional to one project's size, not the total of all projects on the machine.
