# ADR 025: Advisory File Locking for Concurrent Agent Writes

## Status

Accepted

## Context

Multiple AI agents may operate concurrently within the same repository, each executing stories, filing reports, or adding comments. Because the data layer is YAML files on the filesystem (ADR-001), concurrent writes to the same file risk a last-writer-wins race condition where one agent's changes silently overwrite another's. The most frequent collision scenario is two agents adding stories to the same epic file, since stories are embedded arrays within their parent epic (ADR-003).

The system cannot use a database, message queue, or coordination service -- the entire architecture is built on the premise that the filesystem is the only integration point (ADR-008). OS-level `flock()` was considered but rejected because Node.js does not expose POSIX advisory locks natively, the behavior varies across operating systems and network filesystems, and the locking semantics are tied to file descriptors rather than logical operations.

The lock mechanism must be simple enough for agents to reason about, must not require external dependencies, and must degrade gracefully when a process crashes while holding a lock.

## Decision

We will implement advisory file locking in `src/lib/fs.ts` via a `withLock(targetFile, fn)` function that wraps read-modify-write operations on YAML files.

Lock files are named `.pm/.lock-{hash}` where `{hash}` is the first 8 hex characters of a SHA-256 digest of the target file's absolute path. Lock files contain a JSON payload with the acquiring process's PID and a creation timestamp.

Acquisition follows a check-break-acquire sequence. First, `tryAcquireLock` checks whether the lock file exists via `existsSync`. If it exists, the function checks staleness by comparing the lock's creation timestamp against a 30-second wall-clock timeout. Stale locks are forcibly removed via `unlinkSync`, handling crashed processes without requiring PID liveness checks. If the lock exists and is not stale, acquisition fails for this attempt. Finally, `fs.writeFileSync` with the `wx` flag (exclusive creation) atomically creates the lock file, failing if another process won the race between the staleness check and the write. The `wx` flag is the critical correctness mechanism: even when two processes simultaneously detect and break a stale lock, only one succeeds at the `wx` write.

If the `.pm/` directory is not found in the target file's ancestor chain, the lock file falls back to being placed alongside the target file's parent directory.

The retry strategy attempts acquisition up to 3 times with 100ms delays between attempts. If all attempts fail, the operation throws a `PmError` with code `LOCK_ACQUISITION_FAILED`.

Lock release occurs in a `finally` block to ensure cleanup even when the wrapped operation throws. The `unlinkSync` call silently ignores errors to handle races where another process already broke the stale lock.

## Consequences

Story additions and updates, resolution task creation, and other write operations that modify shared YAML files are now protected against concurrent corruption. The `withLock` wrapper is used in `storyAdd`, `storyUpdate`, and `addResolutionStory`.

The locking is advisory, not mandatory. Code paths that bypass `withLock` -- such as the consolidation pipeline's marking phase, which sets `consolidated: true` on reports and comments -- can still race. This is acceptable because marking is idempotent: writing `true` over `true` is harmless.

The 30-second stale timeout means a legitimate long-running operation could have its lock broken by another process. In practice, all locked operations are fast read-modify-write cycles on small YAML files, completing well within the timeout. If operations ever grow to take seconds, the timeout would need adjustment.

Lock files accumulate in `.pm/` during normal operation but are cleaned up on release. A hard crash leaves one orphaned lock file per in-flight operation, which will be broken on the next access after 30 seconds. No periodic cleanup is needed.

There is a narrow TOCTOU (time-of-check-time-of-use) window in the stale lock path: two processes can both observe a stale lock, both remove it, and both attempt the `wx` write. The `wx` flag ensures only one succeeds, so correctness is maintained, but the losing process wastes a retry attempt. In practice, stale locks are rare (they require a process crash) and simultaneous stale-lock races are rarer still.

The SHA-256 path hashing means two files with different absolute paths will almost certainly produce different lock file names, avoiding false contention. The 8-character hex truncation provides 32 bits of entropy, which is sufficient for the expected number of concurrent lock targets in a project management tool.
