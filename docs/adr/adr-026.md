# ADR 026: Multi-Phase Consolidation Pipeline

## Status

Accepted

## Context

AI agents executing stories produce execution reports (structured records of decisions, assumptions, tradeoffs, and potential conflicts) and cross-task comments (asynchronous observations directed at other tasks). Over time, these artifacts accumulate and contain overlapping, contradictory, or complementary information. Without synthesis, the volume of raw artifacts makes it difficult for humans or agents to extract the project's emergent architectural knowledge.

The challenge is to reduce this stream of per-task observations into durable knowledge artifacts -- architecture decision records and resolution tasks for conflicts or gaps -- while handling duplicates, contradictions, and the inherent noise in agent-generated text. A purely rule-based approach cannot handle semantic similarity; a purely LLM-based approach is expensive and slow for the easy cases (exact or near-exact duplicates).

The consolidation pipeline must be idempotent: running it twice on the same input should not produce duplicate outputs. It must also be resumable: if a run processes 10 reports and marks them consolidated, the next run should skip those 10 and process only new arrivals.

## Decision

We will implement a six-phase consolidation pipeline spanning five modules (`consolidate.ts`, `structural-dedup.ts`, `semantic-clustering.ts`, `consolidate-output.ts`, and `src/lib/llm.ts`) that progressively filters and synthesizes agent artifacts.

**Phase 1 -- Ingest.** Load unconsolidated reports from `.pm/reports/` and comments from `.pm/comments/`, filtering out items already marked `consolidated: true` or timestamped before the last consolidation run. Cap reports at `max_reports_per_run` (default 10) to bound per-run cost.

**Phase 2 -- Structural deduplication.** Apply three matching strategies in sequence. First, exact matching on normalized text (lowercase, collapsed whitespace, normalized punctuation) groups identical observations across reports. Second, fuzzy matching via the Dice coefficient on character bigrams (threshold 0.8, minimum text length 20 characters) catches near-duplicates with minor wording differences. Third, contradiction detection identifies opposing decisions across reports by checking negation prefixes ("do not", "avoid", "never", etc.) and comparing the residual object text for similarity (Dice threshold 0.85). Items matched by exact or fuzzy grouping become `confirmed_decision` candidates. Contradictions become conflict pairs. Everything else flows forward as unmatched.

**Phase 3 -- LLM synthesis.** Send only the unmatched items to OpenAI's `gpt-4o-mini` model (temperature 0.3) for classification into confirmed decisions, rejected alternatives, and lessons learned. This phase handles semantic equivalences that structural matching cannot detect. The LLM dependency is isolated behind an `LLMClient` interface in `src/lib/llm.ts`, allowing future model substitution without pipeline changes. If the LLM call fails or returns unparseable output, all items pass through as raw unmatched -- the pipeline degrades gracefully rather than aborting.

**Phase 4 -- Merge.** Concatenate structural candidates and LLM candidates into a single set. The remaining unmatched items (those the LLM could not classify) proceed to clustering.

**Phase 5 -- Route.** Semantic clustering via a second LLM call groups the final residuals by theme and recommends an action per cluster: `create_adr`, `create_task`, or `noop`. The routing phase then creates three types of output artifacts:

- ADRs from `confirmed_decision` candidates, with status `proposed`, author `consolidation-agent`, and tags including `consolidated`, `auto-generated`, and a `dedup:{hash}` tag for idempotency.
- Conflict resolution stories (priority `high`, `resolution_type: "conflict"`) from contradiction pairs detected in Phase 2.
- Gap resolution stories (priority `medium`, `resolution_type: "gap"`) from clusters with a `create_task` recommendation.

All output creation is guarded by SHA-256 content-hash deduplication. The hash of the normalized decision text or task title (truncated to 16 hex characters) is checked against existing ADR tags and story acceptance criteria before creating new artifacts. This ensures idempotency even if the marking phase (below) fails.

The `resolution_type` field on stories is reserved exclusively for the consolidation pipeline. The CLI's `storyAdd` command rejects any attempt to set `resolution_type` directly, enforcing that only the consolidation agent can create conflict and gap tasks. Resolution stories are placed in the first epic with `backlog` status, with a fixed 3 story points.

**Phase 6 -- Mark.** Set `consolidated: true` on every processed report and comment. Update `last_consolidated_at` in `project.yaml` to the current timestamp. This prevents re-processing on subsequent runs.

The pipeline supports `--dry-run` mode, which executes all phases but writes no files, printing proposed actions to the console instead.

## Consequences

The two-tier dedup strategy (structural then LLM) keeps costs low. Exact and fuzzy matches -- typically 60-80% of observations in practice -- are resolved without any API calls. Only genuinely ambiguous items reach the LLM, and the semantic clustering call processes only the final residuals. A typical run with 10 reports makes at most two LLM API calls.

The pipeline depends on an external service (OpenAI) for Phases 3 and 5. If the `OPENAI_API_KEY` is not set, `createLLMClient()` throws immediately, aborting the pipeline before LLM phases execute. The structural dedup and conflict detection phases will have already completed, but their results are lost because the error prevents output routing and marking. To run consolidation without LLM access, the LLM phases would need to be made optional -- this is a known limitation. Within an LLM phase, parse failures of the response JSON are handled gracefully: items pass through as unmatched rather than aborting.

The `gpt-4o-mini` model is hardcoded in `src/lib/llm.ts`. This is a cost/speed optimization -- the model is inexpensive and fast enough for classification tasks -- but it means upgrading the model requires a code change. The `LLMClient` interface makes this a one-line change, but it is not configurable at runtime.

The greedy clustering algorithm in structural dedup is order-dependent: if item A matches both B and C at the fuzzy threshold, but B and C do not match each other, only the first-encountered pair forms a cluster. A union-find structure would produce more optimal clusters but was not worth the added complexity given that hard cases go to the LLM anyway.

The contradiction detection relies on a fixed set of 10 negation prefixes and 9 action prefixes. Subtle contradictions like "use REST" versus "use GraphQL" will not be caught structurally -- they require the LLM phase. This is by design: the structural phase handles the obvious cases cheaply.

The `max_reports_per_run` cap uses naive truncation (first N reports by discovery order) rather than priority-based selection. This means high-value reports are not prioritized over routine ones when the cap is reached. For most projects this is irrelevant because the default cap of 10 exceeds the typical batch size, but large consolidation backlogs could benefit from smarter selection.

Silent `catch` blocks in the ingestion and marking phases mean individual file failures are swallowed. This prevents one malformed report from aborting the entire pipeline, but it also means errors are invisible without `--verbose` output.
