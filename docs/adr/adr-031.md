# ADR 031: Synchronous Filesystem I/O for Simplicity

## Status

Accepted

## Context

Node.js provides both synchronous (`readFileSync`, `writeFileSync`, `existsSync`, `readdirSync`, `renameSync`, `unlinkSync`) and asynchronous (`readFile`, `writeFile`, `readdir`, etc.) filesystem APIs. The async variants are generally recommended for server applications because they avoid blocking the event loop, allowing the process to handle other work while waiting for I/O.

Agent-pm is not a server. It is a CLI tool and MCP server where each invocation handles a single request -- a command from the user, or a tool call from an agent. There is no concurrent request handling within a single process. The MCP server uses stdio transport, processing one tool call at a time. The TUI uses `fs.watch` for live reload but performs reads synchronously during render cycles.

The data files are small YAML documents (typically 1-50KB), stored on a local filesystem. Read and write latencies are sub-millisecond on modern SSDs. The theoretical throughput advantage of async I/O is irrelevant when the total I/O per invocation is a handful of small file operations completing in microseconds.

Using async APIs throughout would require every function that reads or writes YAML to return a `Promise`, propagating `async/await` through the entire call chain. This adds syntactic overhead, complicates error handling (try/catch must wrap awaited calls), and makes the control flow harder to follow in a codebase where operations are inherently sequential.

## Decision

We will use synchronous filesystem APIs (`readFileSync`, `writeFileSync`, `existsSync`, `readdirSync`, `mkdirSync`, `unlinkSync`, `renameSync`) for all YAML I/O operations in the CLI, command implementations, and library code.

The `readYaml` and `writeYaml` helpers in `src/lib/fs.ts` use synchronous reads and writes, providing a clean functional interface (`readYaml(path, schema)` returns the validated object directly, not a Promise). The `withLock` function is the single exception: it is declared `async` to support an async `fn` callback and uses a `setTimeout`-based delay for lock retry, but the lock acquisition itself uses synchronous `writeFileSync` with the `wx` flag.

The MCP server's tool handlers are async (required by the MCP SDK), but they delegate to synchronous CLI invocations via `spawnSync` (ADR-006), so the filesystem I/O within the CLI subprocess is still synchronous.

## Consequences

The codebase is simpler. Functions that read YAML return data directly rather than Promises. Error handling uses straightforward try/catch without `await`. The control flow reads top-to-bottom without interleaving. This matters in a codebase where AI agents (and the consolidation pipeline) may need to understand and modify the code.

The event loop is blocked during file operations. For the CLI, this is invisible because no other work is waiting. For the TUI, synchronous reads during render cycles could theoretically cause frame drops if a YAML file were very large. In practice, with file sizes under 50KB and SSD latencies under 1ms, this has not been observed.

If agent-pm were ever adapted into a long-running server handling concurrent HTTP requests (rather than stdio-based MCP), the synchronous I/O would need to be replaced with async equivalents to avoid blocking. The `readYaml`/`writeYaml` interface would change signature, requiring updates across the codebase. This migration is straightforward but pervasive.

The synchronous `withLock` acquisition means that lock contention causes busy-waiting (100ms sleeps between retries) rather than yielding to the event loop. For the expected contention levels (two or three agents occasionally colliding on the same epic file), this is acceptable. A high-contention scenario with dozens of agents would benefit from an async lock queue, but that scenario exceeds the tool's design parameters.

Network filesystems (NFS, CIFS) have higher latencies than local SSDs. Synchronous I/O on a network mount would block for tens of milliseconds per operation. The local-first architecture (ADR-024) makes this unlikely in practice, but it is a constraint worth noting for CI environments that mount shared storage.
