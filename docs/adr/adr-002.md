# ADR 2: Global Data Directory with PM_HOME Override

## Context

In v1, project data lived inside the `project-management` repository itself, at `./projects/`. This meant the `pm` CLI and all agent workflows only functioned when the working directory was the `project-management` repo. An agent working in a different codebase could not file a story or check project status without first changing directories — a disruptive context switch that broke the autonomous filing workflow.

The core requirement for v2 was that an agent working in _any_ repository on the system should be able to interact with PM data. This demanded a fixed, well-known location for data that does not depend on the current working directory.

Several approaches were considered:

- **XDG Base Directory** (`~/.local/share/pm/`) follows the XDG specification used by many Linux tools. However, macOS does not follow XDG conventions, and `~/.local/share/` is less discoverable than a top-level dotfile directory.

- **A top-level dotfile directory** (`~/.pm/`) is simple, discoverable, and works identically on Linux and macOS. It follows the convention used by tools like `~/.npm/`, `~/.ssh/`, and `~/.config/`.

- **An environment variable only** (no default) would force every user to configure `PM_HOME` before using the tool, adding friction to the initial setup.

The resolution order needed to balance zero-configuration defaults with flexibility for non-standard setups (e.g., CI environments, shared machines, testing).

## Decision

We will store all project data in a global directory that defaults to `~/.pm/projects/`. The `PM_HOME` environment variable, if set, overrides the base directory so that `$PM_HOME/projects/` is used instead. The `getProjectsDir()` function in `src/lib/codes.ts` implements this resolution and never falls back to `process.cwd()`. The directory is created automatically on first CLI invocation via `ensureProjectsDir()`.

## Status

Superseded by [ADR 024](adr-024.md).

The local-first storage model (`.pm/` at the repository root) replaced the global `~/.pm/projects/` directory. The `PM_HOME` environment variable override is preserved for testing but is no longer the primary resolution mechanism. See ADR 024 for the current storage architecture.

## Consequences

The global data directory means `pm` works from any directory immediately after installation. Agents in any repository can call `pm status`, `pm story add`, or any other command without worrying about their working directory. This is the foundation of the autonomous filing workflow.

The `PM_HOME` override enables testing (each test run uses a temporary directory), CI pipelines, and users who want data in a non-standard location. All tests set `PM_HOME` to isolated temporary directories, ensuring they never touch the user's real `~/.pm/` data.

A negative consequence is that project data is now invisible from the repository tree. A developer looking at the `project-management` repo will not see the data unless they know to look in `~/.pm/`. The README and slash commands mitigate this by directing users to `pm status` rather than filesystem exploration.

Another consequence is that the tool now has a hard dependency on the home directory being writable. In containerized or restricted environments where `$HOME` is read-only, `PM_HOME` must be explicitly set to a writable location.
