# ADR 1: YAML Files as the Storage Format

## Context

The `pm` tool needs a persistent storage mechanism for projects, epics, and stories. The primary consumers of this data are AI agents running inside OpenCode and Claude Code, which interact with the world through file reads, file writes, and CLI commands. Human developers are a secondary audience who may want to inspect or hand-edit project state.

Several storage options were considered:

- **SQLite** offers robust querying, transactions, and concurrent access. However, AI agents cannot read or write SQLite databases directly — they would be entirely dependent on the CLI for every interaction. This undermines the design principle that "files are the API."

- **JSON** is machine-readable and universally supported. However, JSON is verbose, does not support comments, and is less pleasant for humans to read or edit. Multi-line strings (descriptions, acceptance criteria) become awkward with escaped newlines. JSON also lacks the block scalar syntax that makes YAML natural for prose-heavy content.

- **A remote database or API** would introduce network dependencies, authentication, and infrastructure requirements that are incompatible with a tool designed to run entirely on a single developer's machine with no external services.

- **YAML** is human-readable, supports multi-line strings natively via block scalars, and is well-understood by AI agents. It can be read and written with standard filesystem operations. The `js-yaml` library is mature and handles round-tripping reliably.

The tension is between query capability (SQLite) and direct agent accessibility (flat files). Since the data volume per project is small — typically dozens of epics with a few stories each — advanced querying is unnecessary.

## Decision

We will use YAML files as the sole storage format for all project management data. Each entity (project, epic) is stored as a separate `.yaml` file in a well-defined directory structure under `.pm/` at the repository root (see [ADR 024](adr-024.md)). The `js-yaml` library handles serialization with `indent: 2`, `lineWidth: 120`, `noRefs: true`, and `sortKeys: false` to produce clean, predictable output.

## Status

Accepted.

## Consequences

YAML's human readability means developers can inspect and debug project state with any text editor or `cat` command, and AI agents can read files directly without going through the CLI. This supports the "files are the API" design principle and means the CLI is a convenience layer, not a gatekeeping layer.

The trade-off is that YAML has no built-in concurrency control. If two agents write to the same epic file simultaneously, one write will silently overwrite the other. In practice this is rare — agents typically work on different stories — but it is a known limitation with no mitigation beyond the single-threaded nature of most agent sessions.

YAML parsing is also more permissive than JSON, which means malformed files can sometimes produce surprising results rather than clean errors. The Zod validation layer (see ADR 6) mitigates this by catching structural issues after parsing.

The choice of YAML over JSON means the project takes a dependency on `js-yaml`, but this is a stable, well-maintained library with no transitive dependencies of concern. The `sortKeys: false` option is important — it preserves the author's intended field ordering, which matters for readability when agents or humans edit files.
