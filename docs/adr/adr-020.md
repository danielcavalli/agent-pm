# ADR 020: Deprecation of @opencode-ai/plugin Custom Tools in Favor of MCP

## Status

Superseded by [ADR 007](adr-007.md)

## Context

pm's initial integration with AI coding clients was through the `@opencode-ai/plugin` package, a custom plugin system specific to OpenCode. This plugin registered pm's tools (status, epic add, story add) as native OpenCode tools, allowing the agent to call them directly. The integration worked — the agent could file stories and check project status within OpenCode — but it created a hard dependency on OpenCode's plugin API.

The `@opencode-ai/plugin` package was an internal OpenCode interface without formal versioning guarantees or cross-client compatibility. Any change to the plugin API would require updating pm's integration code. More critically, the plugin approach was OpenCode-exclusive: users of Claude Code, Cursor, or any other AI coding assistant had no way to use pm's tools programmatically. They could run pm CLI commands manually, but the AI agent in those clients could not invoke pm tools without a compatible integration layer.

When the Model Context Protocol (MCP) emerged as a standardized tool interface supported by multiple clients, it offered a path out of this single-vendor dependency. MCP provides a protocol-level contract: any client that supports MCP can discover and invoke any MCP server's tools, regardless of who built the client or the server. Adopting MCP would make pm's tools available to every MCP-compatible client from a single implementation.

## Decision

We deprecated and removed the `@opencode-ai/plugin` integration in favor of the MCP server described in ADR 007. The `@opencode-ai/plugin` dependency was removed from `package.json`, and all plugin-related code was deleted. The MCP server (`src/mcp-server.ts`) now serves as the sole programmatic interface for AI agents, exposing the same three tools (status, epic add, story add) over the standard MCP protocol via stdio.

## Consequences

The immediate consequence was that pm stopped depending on an OpenCode-internal API. The MCP SDK (`@modelcontextprotocol/sdk`) is a cross-client standard maintained as an open protocol, providing a more stable foundation than a single vendor's plugin system. This decoupled pm's development timeline from OpenCode's plugin API changes.

The migration from custom plugin to MCP was straightforward because pm's tool surface was small (three tools) and their semantics did not change. The tool names, parameter schemas, and descriptions were carried over with minor adjustments to match MCP's JSON Schema format. Existing users experienced no functional change — the same tools were available with the same behavior, just delivered through a different protocol.

The broader consequence was that pm became client-agnostic (see ADR 018). What was previously an OpenCode-only integration became an integration that worked with any MCP-compatible client. This was the primary motivation for the change and is detailed fully in ADR 007.

One minor consequence of removing the plugin was losing OpenCode-native tool UI integration. The custom plugin had access to OpenCode's rendering system and could potentially provide richer tool result formatting (colored output, structured tables, interactive elements). MCP tool results are plain text, so the presentation is limited to what the client does with text content. In practice, this has not been a meaningful loss — the pm tools return human-readable text that both agents and users can interpret without special formatting.

This ADR is retained as a historical record of why the `@opencode-ai/plugin` approach was abandoned. The decision itself is fully superseded by ADR 007, which describes the MCP server that replaced it.
