# ADR 014: Custom Error Hierarchy for Structured CLI Errors

## Status

Accepted

## Context

pm is a CLI tool that needs to communicate errors clearly to two very different audiences: human users reading terminal output, and AI agents parsing tool results. A human needs a concise message explaining what went wrong and, ideally, how to fix it. An AI agent benefits from structured, predictable error formats that it can reason about programmatically — error codes, consistent phrasing, and field-level details for validation failures.

The simplest approach would be to throw plain `Error` instances with descriptive messages and let them propagate to a top-level `catch` that prints the message and exits. This works for small scripts but has two problems at pm's scale. First, it is impossible to distinguish between "expected" errors (file not found, validation failure, invalid project code) and unexpected errors (null reference, network failure, corrupted state) without parsing the error message as a string. Second, different error types carry different contextual data — a validation error has a list of field-level issues, a "not found" error has the entity code that was sought — and a flat string message cannot carry this structured information.

Another approach is to use error codes (numeric or string) without a class hierarchy. Each `throw` site constructs an `Error` with a `code` property, and the top-level handler switches on the code. This is how Node.js core APIs work (`error.code === 'ENOENT'`). It avoids the complexity of a class hierarchy but loses TypeScript's type narrowing: you cannot `catch (err)` and then safely access `err.filePath` without a type assertion, because the catch variable is `unknown`. Each handler must independently check and cast.

A class hierarchy provides both structured error codes and TypeScript type safety. A base `PmError` class carries a `code: string` field, and specific subclasses add context-specific properties: `YamlNotFoundError` has `filePath`, `ZodValidationError` has `zodError` (the Zod error object) and a `fieldErrors` getter, `ProjectNotFoundError` has `projectCode`. The `catch` handler can use `instanceof` to narrow the type and access the specific properties safely.

## Decision

We implement a custom error hierarchy rooted at `PmError`, defined in `src/lib/errors.ts`. `PmError` extends the built-in `Error` class and adds a `code: string` field with values like `YAML_NOT_FOUND`, `YAML_PARSE_ERROR`, `VALIDATION_ERROR`, `PROJECT_NOT_FOUND`, `EPIC_NOT_FOUND`, `STORY_NOT_FOUND`, and `DUPLICATE_PROJECT_CODE`. Each error code has a corresponding subclass that carries the relevant contextual data.

The `action()` wrapper in `cli.ts` provides the top-level error handling. It catches all errors from command handlers: `PmError` instances are formatted with `Error [CODE]: message` in red, while unexpected errors (non-`PmError` instances) are shown with a generic "Unexpected error" prefix and optionally include the stack trace when `--debug` is passed. This two-tier handling ensures that expected errors produce clean, helpful output while unexpected errors surface enough information for debugging without overwhelming the user.

## Consequences

The error hierarchy makes the error handling code in `cli.ts` concise and expressive. The `action()` wrapper is roughly twenty lines of code, and it handles the entire error surface of the application. Adding a new command requires no additional error handling code — the command throws `PmError` subclasses for expected failure cases, and the wrapper formats them consistently. This is a significant improvement over ad-hoc `try/catch` blocks in each command.

The structured error codes benefit AI agent consumption. When the MCP server returns an error (because the underlying `pm` CLI invocation failed), the error message starts with a predictable `Error [YAML_NOT_FOUND]:` or `Error [PROJECT_NOT_FOUND]:` prefix. An AI agent can recognize these patterns and take appropriate action — for example, if it receives `PROJECT_NOT_FOUND` after trying to add a story, it knows to check whether the project code is correct rather than retrying the same operation.

The `ZodValidationError` subclass is particularly valuable. It wraps Zod's `ZodError` object and provides both a human-readable message (with indented field-level details) and a `fieldErrors` getter that returns a structured array of `{ path, message }` objects. This dual representation serves both audiences: the CLI prints the formatted message for humans, while programmatic consumers can access `fieldErrors` for machine-readable diagnostics. The `ValidationError` class serves a similar purpose for non-YAML validation failures (like invalid command-line options), sharing the same error code but originating from different contexts.

One consequence of using `instanceof` checks in the error handler is that the error hierarchy is a runtime contract. If a library function throws a plain `Error` instead of a `PmError`, it will be treated as an unexpected error even if it is a foreseeable failure. This means every expected error path in the codebase must explicitly construct the appropriate `PmError` subclass. In practice, this is straightforward because the error-throwing code is concentrated in a few library functions (`readYaml`, `writeYaml`, and the entity-lookup functions in commands), but it requires discipline — a forgotten `throw new Error(...)` instead of `throw new ProjectNotFoundError(...)` will produce a less helpful error message.

The error codes are strings rather than a TypeScript enum or union type. This means there is no compile-time exhaustiveness check — if you handle `YAML_NOT_FOUND` and `PROJECT_NOT_FOUND` but forget `EPIC_NOT_FOUND`, the compiler will not warn you. In practice, the `action()` wrapper handles all `PmError` instances uniformly (it does not switch on specific codes), so exhaustiveness is not a concern at the top level. But if a future feature needed to handle specific error codes differently, converting the codes to a discriminated union would be worth considering.

The `--debug` flag for unexpected errors is a pragmatic compromise. In normal use, an unexpected error shows a one-line message without a stack trace, keeping the output clean. During development or troubleshooting, `--debug` adds the full stack trace. This two-mode approach balances user experience against debuggability, though it means that bug reports from users may lack the stack trace needed to diagnose the issue. Logging the stack trace to a file (in addition to the conditional console output) would address this, but has not been needed so far.
