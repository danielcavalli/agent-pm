# ADR 010: TypeScript with ESM Modules

## Status

Accepted

## Context

pm is a Node.js CLI tool that needs to handle structured data (YAML files with schemas), interact with the filesystem, manage multiple entry points (CLI, MCP server, TUI), and provide a reasonable developer experience for maintenance and extension. The choice of language and module system affects every layer of the project.

Plain JavaScript was an option. It would have eliminated the compilation step and simplified the build pipeline to nothing â€” just run the source files directly with Node.js. For a small CLI tool, this has real appeal. But pm is not trivially small: it has Zod schemas that define data contracts, a TUI built with React and ink, error hierarchies with typed fields, and library functions that accept generic Zod schema parameters. Without TypeScript's type system, the Zod integration in particular would lose much of its value â€” the `readYaml<S extends z.ZodTypeAny>(filePath: string, schema: S): z.output<S>` function signature in `src/lib/fs.ts` is a generic that infers the return type from the schema parameter, giving callers type-safe access to validated data. In plain JavaScript, every YAML read would return `any`.

Within TypeScript, the choice between CommonJS and ESM modules is significant. CommonJS (`require`/`module.exports`) is the older Node.js standard and has broader compatibility with existing packages. ESM (`import`/`export`) is the JavaScript standard, supported natively by modern Node.js, and is where the ecosystem is moving. Several of pm's dependencies â€” notably `chalk` v5, `ink` v6, and `clipboardy` v5 â€” are ESM-only packages that cannot be `require()`'d from CommonJS. This effectively forces the decision: using these packages means pm must be an ESM project.

The `module` and `moduleResolution` settings in `tsconfig.json` also matter. The `"node16"` setting (as opposed to `"esnext"` or `"bundler"`) requires explicit `.js` extensions in import paths â€” you must write `import { readYaml } from './lib/fs.js'` even though the source file is `fs.ts`. This feels wrong at first (why reference a `.js` file that doesn't exist yet?) but correctly reflects how Node.js ESM resolution works at runtime: Node looks for the compiled `.js` file, not the `.ts` source. Using `"bundler"` module resolution would allow extensionless imports but would create a mismatch between how TypeScript resolves modules and how Node.js does at runtime, leading to confusing failures.

## Decision

pm is written in TypeScript with ESM modules. The `package.json` declares `"type": "module"`, and `tsconfig.json` uses `target: "ES2022"`, `module: "node16"`, and `moduleResolution: "node16"`. All imports use explicit `.js` extensions. The `strict` flag is enabled for full type checking. The `jsx: "react"` option is enabled for the ink-based TUI components.

TypeScript source in `src/` compiles to JavaScript in `dist/`. The build script is `tsc && chmod +x dist/cli.js dist/mcp-server.js` â€” the `chmod` step restores execute permissions on entry points because `tsc` does not preserve the shebang's implicit need for executable permissions. Test files under `src/**/__tests__/**` are excluded from compilation via the `exclude` array in `tsconfig.json`, so they never appear in `dist/`.

## Consequences

TypeScript provides compile-time safety across the codebase. The Zod schema integration is particularly valuable: when `readYaml` is called with `EpicSchema`, the return type is automatically `Epic` (the inferred output type of the schema), and accessing `.stories[0].status` is type-checked against the `StoryStatus` enum. This catches a class of bugs â€” misspelled field names, incorrect status values, missing required fields â€” at compile time rather than at runtime.

The ESM-only stance simplifies the module system. There is no `require()` anywhere, no `__dirname`/`__filename` workarounds (ESM uses `import.meta.url` instead), and no confusion about whether a given file is CJS or ESM. Every file plays by the same rules. This is especially helpful for the TUI code, which uses JSX â€” the `jsx: "react"` compiler option transforms JSX into `React.createElement` calls, and ESM's static imports mean the React dependency is resolved consistently.

The `.js` extension requirement in imports is a minor but persistent friction point. Every new import must include the `.js` extension, and developers unfamiliar with `"node16"` module resolution will initially be confused. IDE support generally handles this well (VS Code adds the extension automatically in auto-imports), but it remains an oddity that requires explanation.

The compilation step means that changes to source files require running `npm run build` before they take effect when running the `pm` binary. During development, this is an extra step between editing and testing. There is no `ts-node` or `tsx` runtime in the production path â€” `ts-node` is a dev dependency used only for running scripts during development. The decision to compile ahead of time rather than using a TypeScript runtime keeps the production startup fast (no on-the-fly transpilation) and ensures that what's deployed is plain JavaScript that Node.js can execute without any TypeScript-specific runtime.

Targeting ES2022 means pm requires Node.js 18 or later (the first LTS version to fully support ES2022 features). This is a reasonable floor for a tool that also depends on recent versions of ink, chalk, and the MCP SDK, all of which have similar or higher Node.js requirements. It does mean pm will not run on older Node.js versions â€” this is an acceptable tradeoff for a development tool where keeping Node.js updated is standard practice.
