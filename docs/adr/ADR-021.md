# ADR 021: v0.1.0-alpha Architecture -- Agent-Aware Supervisory TUI

## Status

Accepted

## Context

### The Vision Gap

agent-pm v0.0.6-alpha has a mature project management data layer (CLI with 13 commands, MCP server with 9 tools, 11 slash commands, Zod-validated YAML storage) and a read-only TUI dashboard. The project's core thesis -- that humans are the bottleneck in AI development, not the agents -- requires the TUI to evolve from a static project board into a live supervisory console where a human can monitor 5-9 AI coding agents from a single terminal pane.

The research (see `docs/research/multi-agent-tui-research.md`) establishes that:
- Human working memory is limited to 3-5 items (Springer cognitive load framework)
- Effective supervisory span of control is 5-9 agents (Gallup, McKinsey)
- Only ~15% of agent decisions need human routing (Anthropic autonomy research)
- ISA-101 industrial display standards define a four-level information hierarchy
- Calm technology principles say the default state should be peripheral, not demanding

### Problems in v0.0.6-alpha That Block a Credible Alpha

**P1: No agent visibility.** The TUI shows project state (epics, stories, status) but has no concept of running agents, their current tasks, or whether they need human attention. There is no mechanism for an agent to register its existence with the system.

**P2: Detail panel does not scroll.** `src/tui/components/DetailPanel.tsx:199` truncates content with `lines.slice(0, height)`. Stories with long descriptions or many acceptance criteria are cut off. This is a basic usability bug -- the adjacent `Tree.tsx` (lines 103-107) already implements a working scroll pattern via `scrollStart = Math.max(0, Math.min(cursor - Math.floor(height / 2), rows.length - height))`.

**P3: No vim-style keybindings.** The TUI uses arrow keys only for navigation. Every major TUI application (lazygit, k9s, btop) uses vim-style j/k/g/G. There is no Tab key for panel focus switching, no `?` for help, and no page-up/page-down. Charm's core design question -- "how can we keep the user from ever wondering what key to press?" -- is unanswered.

**P4: Silent `catch {}` blocks mask errors.** Multiple files swallow errors silently:
- `src/tui/index.tsx:163,172` -- `getPmDir()` and reload errors
- `src/tui/loadTree.ts` -- epic parse errors
- `src/commands/consolidate-output.ts` -- ADR/story creation failures
- `src/commands/epic.ts` -- epicSync and epicList
- `src/commands/gc.ts` -- isTaskCompleted
- `src/commands/status.ts` -- epic loading (note: already has partial handling via `warnOnError` flag at line 108, but other paths are silent)
- `src/commands/work.ts` -- project name lookup

**P5: No concurrency guard.** Multiple agents writing to the same epic YAML file simultaneously can produce corrupted output. The `/pm-work-on-project` orchestrator dispatches stories to parallel sub-agents that may update stories in the same epic.

**P6: Duplicate schema definition.** `TaskReferenceSchema` is defined independently in both `src/schemas/comment.schema.ts` and `src/schemas/adr.schema.ts` with identical logic. This creates maintenance risk if the definitions drift.

**P7: Dead code.** `src/schemas/task-start.schema.ts` defines `ReadyTaskSchema`, `TaskStartQuerySchema`, and `TaskStartResponseSchema`, but none are re-exported from `src/schemas/index.ts`. They appear unused.

**P8: Hardcoded MCP version.** `src/mcp-server.ts:11` hardcodes `"0.0.6-alpha"` rather than reading from `package.json`.

## Decision

### 1. Agent State as Files (Observer Model)

Agents register themselves by writing a heartbeat file to `.pm/agents/{agent-id}.yaml`. The TUI watches this directory via the same `fs.watch` mechanism already used for `.pm/epics/`. The TUI does not spawn, manage, or kill agent processes.

**New schema: `src/schemas/agent-state.schema.ts`**

```yaml
# .pm/agents/{agent-id}.yaml
agent_id: "agent-abc123"
session_id: "session-001"
status: "active"               # active | idle | needs_attention | blocked | completed
current_task: "PM-E001-S003"   # story code, optional
started_at: "2026-03-12T10:00:00Z"
last_heartbeat: "2026-03-12T10:05:30Z"
progress_summary: "Implementing JWT middleware, 3 of 5 acceptance criteria verified"
escalation:                     # optional, present when status = needs_attention
  type: "decision"             # decision | clarification | approval | error
  message: "Should JWT tokens use RS256 or HS256? RS256 is more secure but requires key management."
  confidence: 0.45
  options:
    - "RS256 with key rotation"
    - "HS256 with shared secret"
    - "Let agent decide"
```

**Rationale:**
- Preserves the "files are the API" design principle (ADR-001, ADR-008)
- Works with any agent runtime (Claude Code, OpenCode, tmux scripts, CI)
- No coupling to a specific process management approach
- The existing `useFileWatcher` hook in `src/tui/hooks/useFileWatcher.ts` already handles `.pm/**/*.yaml` -- extending to `.pm/agents/*.yaml` is trivial
- Matches calm technology principles: agents check in periodically, the TUI observes passively
- Research grounding: ISA-101 Level 1 requires knowing who is active; the observer model provides this with minimal infrastructure

**What this defers:** Direct agent spawning/killing from the TUI (requires process lifecycle management, signal handling, output capture, crash recovery). These are v0.2.0+ concerns.

### 2. TUI Three-Panel Layout

The TUI evolves from a two-panel layout (tree + detail) to a three-panel layout with a collapsible agent sidebar:

```
+--Agent Sidebar--+-----Project Tree------+------Detail-------+
| A1 ● working    | > E001 Foundation     | Code: PM-E001-S003|
|   PM-E001-S003  |   ✓ S001 Init TS...  | Title: JWT middle.|
| A2 ▲ ATTENTION  |   ✓ S002 Zod...      | Status: in_progress|
|   PM-E002-S001  | > E002 CLI            | Priority: high    |
| A3 ○ idle       |                       | Points: 3         |
|                 |                       |                   |
|                 |                       | Description:      |
|                 |                       |  [scrollable]     |
+-----------------+-----------------------+-------------------+
| Status bar: j/k nav | Tab focus | a agents | ? help | q quit |
+-------------------------------------------------------------+
```

- **Agent sidebar:** 20-25 characters wide, shows agent_id, status icon, and current_task
- **Collapsible:** Toggle with `a` key. When hidden, restores the existing 40/60 two-panel layout
- **Auto-hide:** If `.pm/agents/` is empty or missing, defaults to the two-panel layout (backward compatible with v0.0.6)
- **Detail panel:** Gains scroll support, replicating the `Tree.tsx` virtual window pattern

**Rationale:**
- Master-detail + sidebar hybrid follows the lazygit multi-panel pattern (proven effective for multi-resource monitoring)
- Collapsible sidebar ensures backward compatibility and works for users not using multi-agent workflows
- ISA-101 Level 1 (fleet overview) maps to the sidebar; L2 (operating) to the tree; L3 (diagnostic) to the detail panel
- Research grounding: IttyBitty's agent list panel directly inspires this layout

### 3. Panel Focus and vim-style Navigation

**Focus management:**
- Tab cycles focus: agent sidebar -> project tree -> detail panel
- Active panel gets a colored border (cyan for active, default for inactive)
- `useFocusManager()` from Ink manages focus state

**Keybindings (normal mode):**

| Key | Action |
|-----|--------|
| j / Down Arrow | Move cursor down in active panel |
| k / Up Arrow | Move cursor up in active panel |
| Tab | Cycle panel focus |
| Enter | Expand/collapse (tree), select (sidebar) |
| f | Cycle status filter (all/backlog/in_progress/done) |
| / | Enter search mode |
| a | Toggle agent sidebar |
| e | Open escalation action (when agent needs attention) |
| ? | Show help overlay |
| c / y | Copy selected code to clipboard |
| q | Quit |
| Escape | Clear filters, exit modes |

**Detail panel scroll:** When the detail panel has focus, j/k scroll the content rather than moving a cursor. This replicates the `scrollStart` pattern from `src/tui/components/Tree.tsx:103-107`.

**Rationale:**
- vim-style j/k is the universal TUI convention (lazygit, k9s, btop, midnight commander)
- Tab panel cycling follows midnight commander and lazygit patterns
- Charm's design principle: "the user should never wonder what key to press"
- `?` for help is standard across vim-inspired TUIs
- Research grounding: clig.dev recommends suggesting next actions; the help overlay serves this purpose

### 4. Escalation via Data Layer

Agents communicate escalation requests by writing to their state file:

```yaml
status: needs_attention
escalation:
  type: decision
  message: "The existing auth middleware uses session cookies. Should I migrate to JWT (breaking change) or add JWT as a parallel path?"
  confidence: 0.42
  options:
    - "Migrate to JWT (breaking change, cleaner long-term)"
    - "Add JWT parallel path (no breaking change, more complexity)"
    - "Skip this story, needs architectural discussion"
```

The TUI displays `needs_attention` agents with a highlighted indicator (red/bold) in the sidebar. When the user navigates to the agent and presses `e`, the detail panel shows the full escalation context with numbered options.

The human's response is written to `.pm/agents/{agent-id}-response.yaml`:

```yaml
agent_id: "agent-abc123"
responded_at: "2026-03-12T10:08:00Z"
selected_option: 1
additional_context: "JWT is the right call. We're deprecating sessions next quarter anyway."
```

The agent polls for responses via the `pm_agent_check_response` MCP tool.

**Rationale:**
- Management by exception: human only engages when an agent explicitly asks (JAPCC)
- Q&A model: agents ask questions when uncertain rather than human approving every action (HBR/Waymo)
- Only ~15% of decisions need human routing (Anthropic autonomy research)
- File-based communication preserves the existing architecture (ADR-001, ADR-008)

### 5. MCP Tool Extensions

Three new tools, all additive (existing 9 tools unchanged):

**`pm_agent_heartbeat`** -- Agent calls this periodically (recommended: every 10-30 seconds)
- Parameters: `agent_id` (required), `session_id`, `status`, `current_task`, `progress_summary`, `workdir`
- Writes/updates `.pm/agents/{agent_id}.yaml`
- Creates `.pm/agents/` directory if it doesn't exist

**`pm_agent_escalate`** -- Agent writes an escalation request
- Parameters: `agent_id` (required), `type` (decision/clarification/approval/error), `message`, `confidence`, `options[]`, `workdir`
- Updates the agent's state file with escalation data and sets status to `needs_attention`

**`pm_agent_check_response`** -- Agent polls for human response
- Parameters: `agent_id` (required), `workdir`
- Returns response content if `.pm/agents/{agent_id}-response.yaml` exists, or "no response"
- Optionally deletes the response file after reading (to prevent re-reading)

**Rationale:**
- MCP tools are the agent-facing API surface (ADR-007)
- Each tool delegates to the CLI via `spawnSync` (ADR-006)
- Heartbeat frequency of 10-30s produces 0.5-1.0 state changes/second with 5 agents -- well within Ink's 32ms throttle and the existing 300ms `useFileWatcher` debounce
- Tools follow the existing pattern: JSON schema input, CLI delegation, text output

### 6. Technical Debt Resolution

**P2 fix -- Detail panel scrolling:** Add `scrollOffset` state to `DetailPanel`. When the detail panel has focus, j/k adjusts `scrollOffset`. Render `lines.slice(scrollOffset, scrollOffset + height)` instead of `lines.slice(0, height)`. Clamp `scrollOffset` to `[0, lines.length - height]`.

**P3 fix -- vim keybindings:** Extend `useInput` in `src/tui/index.tsx` to handle `j` (same as downArrow), `k` (same as upArrow). Add `focusedPanel` state for Tab cycling.

**P4 fix -- Silent catches:** Replace all `catch {}` with `catch (err) { process.stderr.write(...) }` in non-user-facing code, or with proper error state propagation in TUI components.

**P5 fix -- Advisory file locking:** Add a `withLock(filePath, fn)` wrapper in `src/lib/fs.ts` that creates `.pm/.lock-{hash}` files with PID + timestamp. Stale locks (>30s) are broken. This is adequate for <10 concurrent agents.

**P6 fix -- Duplicate TaskReferenceSchema:** Define `TaskReferenceSchema` once in `src/schemas/comment.schema.ts` (which already imports from both `epic.schema.ts` and `story.schema.ts`). Re-export from `src/schemas/index.ts`. Import in `adr.schema.ts` instead of redefining.

**P7 fix -- Dead code:** Either export `ReadyTaskSchema` etc. from `src/schemas/index.ts` and use them in the agent heartbeat flow, or remove `task-start.schema.ts` entirely.

**P8 fix -- MCP version:** Replace `"0.0.6-alpha"` in `src/mcp-server.ts:11` with a `createRequire(import.meta.url)` call to read `package.json`, matching the pattern already used in `src/cli.ts:8-9`.

## Consequences

### Positive

- **Agent visibility without process management.** The observer model gives humans fleet-wide awareness with zero infrastructure beyond the filesystem. Any agent that can write YAML (or call an MCP tool) becomes visible.
- **Backward compatible.** v0.0.6 `.pm/` directories work without any migration. The agent sidebar auto-hides when no agents are registered.
- **Incremental.** Each decision is independently implementable and testable. The three-panel layout can ship without the escalation mechanism; the heartbeat tool can ship without the TUI changes.
- **Research-grounded.** Every design decision traces to specific findings in the research document. The ISA-101 hierarchy, calm technology principles, and management-by-exception pattern have decades of industrial validation.
- **Framework-agnostic.** The file-based observer model works with Claude Code, OpenCode, custom scripts, CI pipelines, or any future agent runtime. No vendor lock-in.

### Negative

- **Polling latency.** Agents discover human responses by polling `pm_agent_check_response`. This adds latency compared to a push-based mechanism (WebSocket, IPC). For coding agent workflows where decisions are non-urgent, this is acceptable.
- **No agent output streaming.** The TUI cannot show real-time agent logs in v0.1.0-alpha. Users must still switch to the agent's terminal for live output. This is the biggest gap compared to IttyBitty and Conduit.
- **File I/O overhead.** Each heartbeat writes a YAML file. With 10 agents at 10s intervals, this is 1 write/second -- negligible on modern SSDs but worth noting.
- **Advisory locking is not bulletproof.** PID-based lock files can be orphaned if an agent is killed with SIGKILL. The 30s staleness timeout mitigates this but doesn't eliminate it.
- **Escalation UX is minimal.** The `e` key + numbered options is functional but not rich. Complex escalations (e.g., showing diffs, asking for code review) will need a more sophisticated interaction model in v0.2.0+.

### What This Enables (v0.2.0+ path)

- Agent spawning from TUI (build on top of heartbeat infrastructure)
- Agent log streaming (add a log file path to agent state, implement tail-view component)
- Token/cost tracking (add token_count fields to heartbeat)
- Confidence-based auto-routing (use escalation confidence scores with calibrated thresholds)
- OpenTelemetry integration (agent state schema can be extended with trace IDs)
