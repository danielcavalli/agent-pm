# ADR 017: AGENTS.md Rules for Autonomous Filing Behavior

## Status

Accepted

## Context

pm's core value proposition is that AI agents file work items — bugs, tech debt, missing tests, feature ideas — as they encounter them during normal coding tasks. Without explicit instructions, an AI agent will not spontaneously decide to call `pm_story_add` when it notices a missing error handler in code it is reviewing. The agent needs to be told what to look for, when to file, when not to file, how to find the right epic, and how to continue its original task afterward.

The two AI coding clients pm targets — OpenCode and Claude Code — both support an AGENTS.md file (or equivalent) that is injected into the agent's system prompt at the start of every conversation. Content in this file shapes the agent's behavior across all interactions, not just when the user explicitly invokes pm commands. This is the mechanism by which pm can make the agent a proactive participant in project management rather than a passive tool executor.

The question was how prescriptive these rules should be. Too vague ("use pm tools when you find issues") and the agent will over-file trivial matters or under-file important ones. Too detailed ("file a story for every function longer than 50 lines") and the rules become brittle, inapplicable to diverse codebases, and annoying when the agent files stories the user does not care about.

There was also a tension between autonomy and control. Filing a story to an existing epic is a low-risk action — the user can review and delete it later. Creating a new project is a high-risk action — it creates directory structure and could represent a misunderstanding of the user's intent. The rules need to draw a line between actions the agent can take independently and actions that require user confirmation.

## Decision

We define a structured set of rules in `install/agents-rules.md` that are injected into the project's `AGENTS.md` file via `pm rules init`. (Originally these were injected globally by the installer per ADR 016; they now use per-project injection to align with the local-first storage model in ADR 024.) The rules cover four areas: when to file (discovering bugs, tech debt, missing features, inadequate tests, or performance concerns that are unrelated to the current task), when not to file (the issue is part of the current task, is trivial enough to fix immediately, or the agent is unsure whether it is actually a problem), how to file (check existing projects and epics first via `pm_status`, then use `pm_story_add` or `pm_epic_add` with clear titles and descriptions), and an explicit prohibition against using filesystem commands to read project data directly.

The rules instruct the agent to always continue its current task after filing — filing is a side effect, not a context switch. They also require the agent to notify the user before creating a new project, recognizing that project creation is a higher-impact action than adding a story to an existing epic.

## Consequences

The rules enable the autonomous filing behavior that makes pm more than just a passive CLI tool. An AI agent working on a feature implementation that encounters an unrelated bug can now file a story for that bug, assign it to the appropriate epic, and continue working on the original feature. Without these rules, the bug would either be fixed ad hoc (scope creep), ignored (lost knowledge), or require the user to manually file it later (interruption).

The "when not to file" section is as important as the "when to file" section. Without it, agents tend to be overly enthusiastic about filing, creating stories for every minor observation. The two-minute rule ("if it's trivial and can be fixed in under 2 minutes, just fix it") and the uncertainty rule ("if you're unsure whether it's actually a problem, mention it to the user instead") prevent noise in the backlog. These rules are based on observed agent behavior during development — early versions without these constraints produced backlogs full of single-line typo fixes and speculative performance concerns.

The prohibition against filesystem exploration (`NEVER use find, grep, ls, or other filesystem commands to discover or read project, epic, or story data`) addresses a specific failure mode observed in practice. Without this rule, agents would sometimes bypass the CLI and directly read YAML files to understand project state. This is problematic for two reasons: it bypasses Zod validation (the agent could read and act on invalid data), and it creates brittle workflows that depend on the internal file structure (breaking if the storage format changes). By routing all reads through `pm_status` and the CLI, the rules ensure the agent uses the same validated data path as every other interface.

The rules are static text, not code. They cannot adapt to the specific project, codebase, or team conventions the user is working with. A user who never wants agents to file performance-related stories has no way to exclude that category short of editing the rules file manually. A more dynamic approach — configuration flags that toggle which categories of issues trigger filing — would provide more user control but would complicate both the rules and the installer. For now, the static rules represent a reasonable default that works for most single-developer workflows.

One limitation is that the rules' effectiveness depends entirely on the AI agent's compliance. Different agent models may interpret the rules more or less faithfully. A model with weak instruction-following might ignore the "when not to file" constraints and produce excessive filings, while a model that is too conservative might never file autonomously even when the rules clearly apply. The rules are written to be unambiguous — using concrete examples and explicit thresholds — but there is no enforcement mechanism beyond the agent's adherence to its system prompt.

The rules also encode a specific workflow assumption: that the user has already set up projects and epics, and the agent is primarily filing stories. If a user has no projects configured, the agent will run `pm_status`, see no results, and (per the rules) notify the user rather than creating a project autonomously. This is a deliberate safety measure, but it means the agent cannot bootstrap pm usage from scratch without user involvement.
