# ADR 018: Dual-Client Support from Single Codebase

## Status

Accepted

## Context

pm targets two AI coding clients: OpenCode and Claude Code. Both support MCP tool integration, slash commands, and agent instruction files, but they differ in configuration format, directory locations, and minor behavioral details. The question was whether to maintain a single pm codebase that serves both clients or to fork into client-specific distributions.

The differences between the two clients are superficial but pervasive. MCP server registration uses `~/.config/opencode/opencode.json` for OpenCode (with an `mcp` key whose values contain `command` as an array) and `~/.claude/settings.json` for Claude Code (with an `mcpServers` key whose values separate `command` and `args`). Slash commands go to `~/.config/opencode/commands/` versus `~/.claude/commands/`. Agent rules go to `~/.config/opencode/AGENTS.md` versus `~/.claude/AGENTS.md`. These are all path and JSON structure differences — the content being installed is identical.

A fork-per-client approach would mean two installers, two sets of slash commands (possibly diverging over time), two AGENTS.md rule files, and two sets of documentation. The MCP server and CLI would remain shared, since they have no client-specific code, but the integration layer would double in maintenance cost. Any improvement to a slash command would need to be ported to both forks, and testing would require verifying both installations.

The alternative — a single codebase with client-aware installation — keeps all the shared content in one place and uses the installer to handle the differences. The cost is that the installer itself becomes more complex, with conditionals for each client. But the installer is already a single Bash script, and the conditionals amount to a few if/else blocks.

## Decision

pm maintains a single codebase with a single installer that detects and configures both clients. All content — MCP server, slash commands, agent rules — exists in one canonical location within the repository. The installer handles client-specific differences through per-client functions (`configure_mcp_opencode`, `configure_mcp_claude`, `install_commands`, `install_rules`) that map the shared content to each client's expected locations and formats.

The MCP server itself is entirely client-agnostic. It communicates over stdio using the MCP protocol, and neither the server code nor the tool implementations contain any client-specific logic. The same `dist/mcp-server.js` file is registered in both clients' configurations.

Slash command markdown files use the `$ARGUMENTS` placeholder for parameter injection, which is supported by both OpenCode and Claude Code with the same semantics. No client-specific slash command variants exist.

## Consequences

The single-codebase approach eliminates the risk of client-specific drift. When a slash command is updated (for example, improving the `/pm-work-on` workflow instructions), the change is made once in `install/commands/pm-work-on.md` and applied to both clients on the next installer run. There is no possibility of one client receiving an update while the other does not, which would be a real risk with forked distributions.

The installer's client detection and configuration logic is concentrated in one file (`install/install.sh`), with a clear comment block at the bottom documenting the differences between clients. This serves as both the implementation and the documentation of what varies between clients. A developer adding support for a third client (say, Cursor or Windsurf, if they adopt MCP) would add a new detection check, a new `configure_mcp_*` function, and entries in the command and rules installation sections — all following the established pattern.

The tradeoff is that the installer must understand each client's configuration format. The inline `node -e` scripts that edit JSON configuration files contain client-specific JSON paths (`config.mcp['pm-tools']` vs `config.mcpServers['pm-tools']`) and object structures (`{ type: 'local', command: [...] }` vs `{ command: '...', args: [...] }`). If a client changes its configuration format, the installer must be updated. This coupling is unavoidable — any integration with an external tool depends on that tool's configuration interface — but it means the installer must be tested against both clients after any change.

The slash command format compatibility between clients is a fortunate coincidence rather than a guaranteed standard. Both OpenCode and Claude Code currently interpret `$ARGUMENTS` the same way, but there is no formal specification that they must continue to do so. If one client changed its variable substitution syntax, pm would need client-specific command variants — the first crack in the single-content model. Monitoring client release notes for changes to slash command handling is a minor but ongoing maintenance task.

The client-agnostic MCP server is the strongest point of convergence. Because MCP is a standardized protocol, the server implementation genuinely does not need to know which client is calling it. This is a direct benefit of adopting MCP (ADR 007) — the protocol abstraction makes the dual-client support almost free at the tool layer. The only client-specific work is in the configuration layer (how the server is registered), which is handled by the installer.

One practical limitation is that testing the full integration requires having both clients installed. The CI pipeline can verify the CLI and MCP server in isolation, but verifying that slash commands work correctly, that agent rules are properly injected, and that MCP tools are callable from each client requires manual testing with both clients. The `install.sh` comment block explicitly notes this: "full live verification of Claude Code integration requires the `claude` CLI to be installed."
