# ADR 011: Zod for Runtime Validation at the YAML Boundary

## Status

Accepted

## Context

pm stores all its data as YAML files on disk. These files are the single source of truth (ADR 001) and are read by every interface — CLI, MCP server, slash commands, and TUI (ADR 008). Unlike a database with an enforced schema, YAML files on disk have no inherent structure guarantees. A user could manually edit a file and introduce a typo in a status field, delete a required property, or add a story with an invalid code format. An AI agent could write malformed data if the CLI had a bug. A migration from v1 to v2 format could leave files in an inconsistent state.

The question is how to defend against invalid data entering the system or corrupting the read path. There are several approaches. One is to validate data at write time only, trusting that anything already on disk is correct. This is fragile because it cannot catch manual edits or migration artifacts. Another is to validate at both read and write time, treating every YAML file access as a boundary crossing that requires verification. A third approach is to use TypeScript interfaces to define the data shape and rely on the type system alone, but TypeScript types are erased at runtime and provide zero protection against malformed files.

Zod is a TypeScript-first schema validation library. It defines schemas as runtime values (not just types) that can parse and validate data, throwing structured errors on failure. Critically, Zod schemas can infer TypeScript types from their definitions, so a single schema serves as both the runtime validator and the compile-time type definition. This eliminates the common problem of type definitions drifting from validation logic.

Other validation libraries exist. Joi is mature and widely used but predates TypeScript and requires separate type definitions. Ajv validates JSON Schema and is very fast but operates on JSON Schema objects rather than TypeScript-native definitions, making it verbose for complex patterns. Yup is similar to Zod but has weaker TypeScript inference and a larger API surface. io-ts integrates deeply with functional programming patterns (Either types, codecs) that would add conceptual overhead without clear benefit for a CLI tool.

## Decision

We use Zod schemas as the authoritative definition of pm's data structures. Every YAML file is validated against its corresponding Zod schema on every read. The `readYaml` function in `src/lib/fs.ts` takes a Zod schema as a parameter and returns the schema's output type, combining file reading, YAML parsing, and schema validation into a single operation that either succeeds with typed data or throws a structured error.

Schemas are defined in `src/schemas/` — one file per entity (`project.schema.ts`, `epic.schema.ts`, `story.schema.ts`). They define field types, enum constraints, regex patterns for codes and IDs, default values for optional fields, and cross-field validation via Zod's `superRefine` method. The TypeScript types used throughout the codebase (`Story`, `Epic`, `Project`) are inferred from these schemas using `z.infer<typeof Schema>`, ensuring that the types and the validation logic are always in sync.

## Consequences

Validating on every read means that malformed data is detected immediately, regardless of how it was introduced. If someone hand-edits an epic file and misspells `in_progress` as `inprogress`, the next `pm status` or `pm story list` command will throw a `ZodValidationError` with a clear message: `stories.0.status: Invalid enum value. Expected 'backlog' | 'in_progress' | 'done' | 'cancelled', received 'inprogress'`. This error surfaces the exact field, the expected values, and the actual value, making it straightforward to diagnose and fix.

The `readYaml<S extends z.ZodTypeAny>(filePath: string, schema: S): z.output<S>` function signature is the linchpin of the type-safety chain. When a command calls `readYaml(epicPath, EpicSchema)`, the return type is `Epic` — a fully typed object with `.stories` as `Story[]`, `.status` as `EpicStatus`, and so on. The caller does not need a type assertion, a cast, or a separate type declaration. This propagates type safety from the schema definition all the way through command logic to output formatting, with no gaps.

The Zod schemas also serve as living documentation of the data model. Reading `story.schema.ts` tells you that story codes must match `^[A-Z]{2,6}-E\d{3}-S\d{3}$`, that story points must be one of `[1, 2, 3, 5, 8]`, that status must be one of four enum values, and that `acceptance_criteria` defaults to an empty array if omitted. This is more precise and more trustworthy than prose documentation because it is enforced at runtime.

Validating on every read does introduce a performance cost. Every `readYaml` call parses the YAML string, then runs it through Zod's validation pipeline, which checks types, regex patterns, enum membership, and custom refinements. For pm's file sizes (typically a few KB per epic file, with a handful of stories each), this overhead is negligible — microseconds per file. But it scales linearly with the number of files and stories, so a project with thousands of epics would feel this cost in commands like `pm status` that read every file. pm is designed for small-to-medium projects (tens of epics, hundreds of stories), so this has not been a practical concern.

One tradeoff is that Zod is a runtime dependency that adds to the package size. It is roughly 60KB minified, which is modest but non-trivial for a CLI tool. The alternative — hand-writing validation functions — would eliminate this dependency but would require maintaining validation logic separately from type definitions, a duplication that Zod's type inference eliminates. The dependency cost is justified by the reduction in maintenance burden and the quality of error messages.

Another consideration is that Zod's `superRefine` method, used for cross-field validation in the epic schema (ensuring story codes are consistent with the epic's code), produces less readable code than simple field-level validators. These refinement functions are the most complex part of the schema definitions and require understanding Zod's `ctx.addIssue` API. For pm's relatively simple data model, this complexity is manageable, but it represents a learning curve for contributors unfamiliar with Zod's advanced features.
