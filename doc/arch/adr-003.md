# ADR 3: Stories Embedded Inside Epic YAML Files

## Context

The data model has three levels: Projects, Epics, and Stories. Projects are directories. Epics need to be individual files (for independent creation and reading). The question is where stories live.

Two approaches were considered:

- **Stories as separate files** (e.g., `epics/E001/S001.yaml`, `epics/E001/S002.yaml`). This would give each story its own file, enabling independent reads and writes. However, it would create a deeply nested directory structure, make it harder to see all stories for an epic at a glance, and require reading many small files to assemble an epic's full state. For a project with 110 stories across 29 epics, that would mean 139 separate YAML files instead of 29.

- **Stories embedded in epic files** (e.g., `epics/E001-foundation.yaml` contains a `stories:` array). This keeps each epic self-contained — one file to read, one file to understand. Adding a story means appending to an array in the parent epic. Listing stories means reading one file.

The tension is between write granularity (separate files are safer for concurrent edits) and read coherence (embedded stories give you the full epic context in one read). Since AI agents typically work on one story at a time within one epic, and the number of stories per epic is small (typically 2-8), the concurrent write risk is low.

## Decision

We will embed stories as a YAML array inside their parent epic file. Each epic file (`E{NNN}-{slug}.yaml`) contains a top-level `stories:` key with an array of story objects. Stories are identified by their `id` (e.g., `S001`) and `code` (e.g., `PM-E001-S001`) within the array.

## Status

Accepted.

## Consequences

An epic file is a complete, self-contained unit. Reading one file gives you the epic's metadata and all its stories, acceptance criteria, and statuses. This makes the `pm work` command simple — parse one file, find the story, print context. It also means `pm status` can assemble a full project view by reading one file per epic rather than one file per story.

The trade-off is that updating a single story requires reading the entire epic, modifying the stories array, and writing the entire file back. For an epic with 7 stories, this means rewriting ~100 lines of YAML to change one status field. This is a minor inefficiency with no practical impact at the expected data scale, but it does mean two agents updating different stories in the same epic simultaneously would conflict.

A Zod `superRefine` validator on the epic schema enforces story ID uniqueness within an epic, preventing accidental duplicates when stories are appended programmatically.

The embedded approach also means there is no way to reference a story file directly by path — you must know the epic code to find the file, then search the stories array. The `parseStoryCode()` and `findEpicFile()` helpers in `src/lib/codes.ts` encapsulate this indirection.
