# ADR 007: MCP Server as Universal Tool Interface

## Status

Accepted (supersedes ADR 020)

## Context

pm was originally built as a CLI tool with an additional integration layer specific to OpenCode: a custom plugin using the `@opencode-ai/plugin` package. This plugin registered pm's capabilities as OpenCode-native tools, allowing the AI agent within OpenCode to invoke pm operations directly. The approach worked, but it locked pm into a single client. Anyone using Claude Code, Cursor, Windsurf, or another AI coding assistant had no way to integrate pm's tools into their workflow without manual CLI invocation.

The Model Context Protocol (MCP) emerged as a standardized way for AI coding clients to discover and invoke external tools. MCP defines a JSON-RPC-based protocol over stdio (or HTTP with SSE) where a server advertises tool schemas and handles invocations. Both OpenCode and Claude Code support MCP natively: they can start an MCP server as a subprocess, query its tool list, and call those tools on behalf of the AI agent. This means a single MCP server implementation can serve any compatible client without client-specific code.

The question was whether to maintain the existing OpenCode plugin alongside a new MCP server, adopt MCP exclusively, or build some other abstraction layer. Maintaining both would mean two integration paths to test and keep in sync. Building a custom abstraction would add complexity without any standard to lean on. Adopting MCP exclusively would require dropping the existing plugin but would gain compatibility with every MCP-supporting client, present and future.

There was also a design question about how the MCP server should execute operations. It could import pm's library functions directly (sharing a process with the calling client's MCP runtime), or it could delegate to the CLI as a subprocess. ADR 006 addresses this in detail — the decision was to delegate via `spawnSync`.

## Decision

We replace the `@opencode-ai/plugin` custom tool integration with a single MCP server (`src/mcp-server.ts`) that exposes three tools — `pm_status`, `pm_epic_add`, and `pm_story_add` — over stdio using the standard MCP protocol. The `@opencode-ai/plugin` dependency is removed entirely. The MCP server is the sole programmatic interface for AI agents to interact with pm.

The server uses the official `@modelcontextprotocol/sdk` package and communicates via `StdioServerTransport`. Each tool defines a JSON Schema for its input parameters, including required fields, enums for constrained values (like priority levels and story point values), and descriptive text that helps the AI agent understand when and how to use each tool. Tool descriptions are written as instructions to the agent, not as API documentation — for example, `pm_story_add` says "Use this when you discover a specific, actionable piece of work while working on something else."

## Consequences

The immediate benefit is client independence. pm's tools are now available in OpenCode, Claude Code, and any future client that supports MCP, without any client-specific code in pm's codebase. The installer detects which clients are present and registers the same MCP server in each one's configuration, differing only in the JSON structure each client expects for MCP server registration (OpenCode uses `mcp.pm-tools.command`, Claude Code uses `mcpServers.pm-tools`).

The tool surface is deliberately small — three tools instead of exposing every CLI subcommand. This is an intentional constraint. AI agents work best with a focused set of tools that have clear, non-overlapping purposes. `pm_status` is for reading, `pm_epic_add` is for creating high-level work themes, and `pm_story_add` is for filing specific tasks. Operations like updating story status, syncing epics, or running migrations are available through the CLI but not exposed as MCP tools, because they are performed by slash commands or human operators rather than autonomously by an agent during normal coding work.

Dropping the `@opencode-ai/plugin` dependency simplifies the build and removes a coupling to OpenCode's internal plugin API, which was not versioned with strong stability guarantees. The MCP SDK, by contrast, follows the MCP specification and is maintained by Anthropic as a cross-client standard.

One tradeoff is that MCP over stdio requires the client to manage the server process lifecycle. If the MCP server crashes, the client must decide whether to restart it. In practice, pm's MCP server is stateless and trivially restartable — it holds no in-memory state between tool calls — so crashes are recoverable by the client's standard MCP process management. However, diagnosing MCP server issues can be awkward because stderr output from the server process may not be surfaced to the user in all clients.

Another consequence is that the tool schemas in `mcp-server.ts` are defined inline as plain objects rather than being generated from the Zod schemas used elsewhere in the codebase. This means the MCP input schemas and the YAML validation schemas could theoretically drift apart. In practice, the MCP schemas are simpler (they define parameter types and enums) while the Zod schemas handle deeper validation (regex patterns, cross-field constraints via `superRefine`). The CLI sits between them: it receives MCP parameters as CLI flags and then validates them through the Zod schemas before writing YAML. So the Zod schemas remain the authoritative validation layer regardless of what the MCP schemas accept.
