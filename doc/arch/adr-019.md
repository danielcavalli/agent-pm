# ADR 019: Migration from Per-Repository to Global Data Directory

## Status

Accepted

## Context

pm originally stored project data in a `projects/` directory within the pm repository itself. This was convenient during early development — the data was versioned alongside the code, and there was no need to manage external file paths. But it had a fundamental limitation: the project data was coupled to the repository checkout. If you cloned the repo elsewhere, the data was not available. If you deleted the repo, you lost your project data. And the MCP server, which runs as a global service, had no reliable way to find the data directory because it did not know where the repo was checked out.

ADR 002 established `~/.pm/` as the global data directory, resolved via the `PM_HOME` environment variable (defaulting to `~/.pm`). This gave pm a stable, repository-independent location for project data. But existing users had data in the old `./projects/` layout that needed to be moved to the new `~/.pm/projects/` location.

The migration could be handled in several ways. An automatic migration on first run would detect the old layout and silently move files, but this is risky — moving files without explicit user consent can lead to confusion, and if something goes wrong, the user may not realize what happened. A flag-day approach (just switch the default and let users figure it out) would be simpler to implement but hostile to users who do not read changelogs. An explicit migration command gives the user control over when and how the migration happens, lets them verify the results, and keeps the process reversible.

## Decision

We provide an explicit `pm migrate` command that copies project directories from a source location (defaulting to `./projects/`) to the global data directory (`~/.pm/projects/`). The migration is a copy, not a move — source files are left in place so the user can verify the result before manually cleaning up. Projects that already exist in the target directory are skipped (not overwritten), ensuring that running the migration multiple times is safe. After copying, the command rebuilds the global index file.

The command accepts a `--source` flag to specify a non-default source directory, allowing migration from any location.

## Consequences

The explicit migration command respects the user's agency. Rather than silently relocating data, it requires the user to invoke `pm migrate`, see the results (which projects were copied, which were skipped), and decide when to delete the old files. This is especially important because the migration is a one-way architectural change — once data lives in `~/.pm/`, the old per-repository layout is no longer supported by any pm command.

The copy-not-move semantics make the migration non-destructive. If the migration produces unexpected results (a project directory was not copied correctly, or a file was corrupted during the copy), the original data is still intact at the source location. The user can fix the issue and re-run the migration, or manually copy individual files. This safety net is worth the small cost of temporary disk space duplication.

The skip-if-exists behavior prevents accidental data loss from repeated migration runs. If a user runs `pm migrate`, makes changes to a project in `~/.pm/`, and then runs `pm migrate` again (perhaps forgetting they already migrated), the existing data in `~/.pm/` is preserved. The command prints a warning for each skipped project so the user understands why some projects were not copied.

The `rebuildIndex()` call at the end of migration ensures the global index is consistent with the newly copied data. Without this step, `pm status` would not see the migrated projects until the next operation that triggers an index rebuild. This is a small detail but important for the user experience — after running `pm migrate`, the user expects `pm status` to immediately show all their projects.

One limitation is that the migration only copies files — it does not transform them. If the data format changed between versions (which has not happened yet but could in the future), a separate format migration would be needed. The `pm migrate` command is specifically about relocating data from the per-repository layout to the global directory; it is not a general-purpose data migration framework.

The migration command will become unnecessary once all existing installations have migrated. It is effectively a transitional tool — important for the v1-to-v2 upgrade path, but irrelevant for new installations that start with `~/.pm/` from the beginning. The command remains in the codebase as it costs nothing to maintain and could be useful if pm ever needs to support another storage location change.
