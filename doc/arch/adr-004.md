# ADR 4: Auto-Rebuilt Index File

## Context

The `pm status` command (with no project argument) needs to show a summary of all projects — name, status, epic count, story count, and completion progress. Loading this information requires reading every `project.yaml` and every epic file across all projects. For a small number of projects this is fast, but the design should anticipate growth.

An index file at `projects/index.yaml` could serve as a cache of aggregate data, avoiding the need to scan all files on every status query. The question is how to keep this index consistent with the underlying data.

Three approaches were considered:

- **Event-driven updates** using file watchers to detect changes and update the index in real time. This adds complexity (watcher lifecycle management, edge cases around rapid writes) and is fragile across different operating systems.

- **Database-backed index** using SQLite or similar. This provides transactional consistency but contradicts the YAML-files-as-storage decision (ADR 1) and requires a separate synchronization mechanism.

- **Rebuild on mutation** — every CLI command that modifies data (init, epic add, story add, story update) calls `rebuildIndex()` after the mutation completes. The index is a derived artifact, always reconstructable from the source YAML files.

## Decision

We will maintain an auto-rebuilt `index.yaml` file at `$PM_HOME/projects/index.yaml`. The `rebuildIndex()` function in `src/lib/index.ts` scans all project directories, reads each `project.yaml` and all epic files, and writes a fresh index. It is called after every mutating CLI command. An optional `filterCode` parameter enables incremental rebuilds that only re-scan one project and merge the result into the existing index.

## Status

Accepted.

## Consequences

The index is always consistent with the underlying data after any CLI-mediated mutation. There is no stale cache problem as long as changes go through the CLI. The `rebuildIndex()` function is simple — about 70 lines — and runs in milliseconds for the expected data scale.

The negative consequence is that changes made by directly editing YAML files (bypassing the CLI) will not be reflected in the index until the next CLI mutation triggers a rebuild. This is acceptable because the index is a convenience for the summary view — the `pm status <PROJECT>` detailed view reads live data directly, not the index. The PRD's design principle that "files are the API" means direct edits are a supported workflow, and the index gracefully catches up on the next mutation.

The incremental rebuild (via `filterCode`) is an optimization for the common case where a single project changes. It avoids re-scanning all projects when only one was modified. The full rebuild remains available as a fallback and is used by `pm migrate`.

Another consequence is that the index file is a derived artifact and should not be hand-edited. It is overwritten completely on each rebuild. This is documented implicitly by the auto-maintenance behavior but could surprise a developer who tries to edit it manually.
