# ADR 015: Global npm Install as Distribution Mechanism

## Status

Accepted

## Context

pm needs to be available as a command-line binary (`pm`) that can be invoked from any directory on the system. This is essential because the MCP server calls `spawnSync("pm", cliArgs)` (ADR 006), slash commands instruct the agent to run `pm` subcommands (ADR 009), and human users type `pm status` in their terminal. The binary must be on the system PATH and must resolve to the correct version of the compiled JavaScript.

There are several ways to distribute a Node.js CLI tool. Publishing to the npm registry and having users run `npm install -g project-management` is the standard approach for public packages. For a tool that is installed from a local repository (not yet published to npm), `npm install -g .` from the repository root achieves the same effect: it links the package globally, creating a `pm` symlink in the global `node_modules/.bin/` directory that points to the file specified in the `bin` field of `package.json`.

An alternative would be to compile pm into a standalone binary using a tool like `pkg`, `nexe`, or `bun build --compile`. This would produce a single executable file with no Node.js runtime dependency — users would download the binary and place it on their PATH. Standalone binaries are attractive for distribution (no Node.js version concerns, no `node_modules`), but they add build complexity, increase binary size (the entire Node.js runtime is bundled), and can cause issues with native modules or dynamic imports. pm uses dynamic `import()` for lazy-loading commands (ADR 012), which may not work correctly in all bundlers.

Another option is to avoid global installation entirely and run pm via `npx project-management` or a local `node_modules/.bin/pm` invocation. This avoids polluting the global namespace but makes the binary harder to find from the MCP server (which needs a stable path) and less ergonomic for human users. The MCP server's `spawnSync("pm", ...)` call assumes `pm` is on the PATH; changing this to a full path would couple the MCP server to the installation location.

## Decision

pm is distributed via `npm install -g .` from the repository root. The `package.json` declares a `bin` field mapping `"pm"` to `"./dist/cli.js"`, and the entry point (`dist/cli.js`) includes a `#!/usr/bin/env node` shebang line that allows it to be executed directly. The build script (`tsc && chmod +x dist/cli.js dist/mcp-server.js`) compiles TypeScript and restores execute permissions on the entry points, since `tsc` outputs regular files without the execute bit.

The MCP server is also installed globally as a side effect — it lives in the same package at `dist/mcp-server.js`. The installer resolves its absolute path via `npm root -g` to find the global `node_modules` directory, then constructs the full path to `dist/mcp-server.js` for MCP server registration.

## Consequences

Global npm install provides a well-understood deployment model. After running `npm install -g .`, the `pm` command is available system-wide, and `which pm` returns a predictable path. The MCP server's `spawnSync("pm", ...)` call works because the global `node_modules/.bin/` directory is on the PATH in standard Node.js installations. This simplicity is the primary benefit — there is no custom installer, no binary packaging, and no PATH manipulation required.

The `chmod +x` step in the build script is a workaround for a TypeScript compiler limitation: `tsc` does not preserve or set file permissions based on shebang lines. Without `chmod`, the globally installed `pm` command would fail with a "permission denied" error on Unix-like systems. This is a small but essential detail that is easy to forget when modifying the build script. If the build were extended (for example, adding a `tsc --watch` mode for development), the `chmod` step would need to be re-applied after each rebuild.

Global npm installation creates a dependency on Node.js being installed. Anyone using pm must have Node.js 18+ and npm available. For the target audience — developers using AI coding assistants — this is a reasonable assumption, since both OpenCode and Claude Code are JavaScript/TypeScript tools that require Node.js. But it does mean pm cannot be used in environments where Node.js is not available or not desired.

The global install also means that only one version of pm can be active at a time per system. If a user works on multiple projects that require different pm versions (an unlikely scenario today but possible in the future), they would need version managers like `nvm` or `volta` to switch between versions. This is a standard limitation of globally-installed npm packages and is not specific to pm.

One operational consideration is that updates require rebuilding and reinstalling: `npm run build && npm install -g .`. There is no auto-update mechanism, no version check, and no notification when a new version is available. For a personal productivity tool installed from a local repository, this is acceptable — the user controls when they update. If pm were published to npm as a public package, the standard `npm update -g project-management` flow would work automatically.
