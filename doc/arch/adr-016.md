# ADR 016: Idempotent Multi-Client Installer

## Status

Accepted

## Context

pm integrates with AI coding clients at three touch points: MCP server registration (so the client can offer pm's tools to its agent), slash command installation (so the user can invoke `/pm-work-on` and similar workflows), and agent rules injection (so the agent knows when and how to file work items autonomously). Each touch point involves modifying a client-specific configuration file or directory. OpenCode and Claude Code store these configurations in different locations and formats.

A manual setup process would require the user to edit JSON configuration files to register the MCP server, copy markdown files to the right directories for slash commands, and append rules to an AGENTS.md file. This is error-prone — a misplaced comma in a JSON file breaks the client, copying commands to the wrong directory makes them invisible, and manually editing AGENTS.md risks corrupting existing content. Worse, the setup must be repeated for each client, and must be re-done whenever pm is updated.

The question was how to automate this setup. One option was a Node.js script (matching pm's implementation language) that could use the same libraries and share code with the main project. Another was a shell script, which is the traditional choice for Unix installation tasks. A shell script has the advantage of being executable without any runtime dependency — it can run even if `npm install` has not yet completed, which is useful because the script's first step is `npm install -g .`.

The installer must also be safe to run repeatedly. A developer who adds a new slash command to pm and re-runs the installer should get the updated command without duplicating existing entries. An installer that appends MCP configuration on every run would create duplicate entries; one that appends agent rules would create multiple copies of the same rules block.

## Decision

We implement the installer as a Bash script (`install/install.sh`) that performs four operations in sequence: globally install the pm CLI via `npm install -g .`, detect which AI coding clients are present, configure each detected client (MCP server registration, slash command copying, agent rules installation), and print a summary.

The script is idempotent at every step. MCP server registration uses inline `node -e` scripts that parse the client's JSON configuration, set the `pm-tools` key (overwriting any existing value), and write the file back. Slash command installation uses `cp -v` to copy files from `install/commands/` to the client's commands directory, overwriting existing files with the same names. Agent rules installation uses `awk` with start and end markers (`# PM Autonomous Filing Rules` / `# END PM Autonomous Filing Rules`) to find and replace the existing rules section, or append it if not present.

Client detection is based on directory existence: `~/.config/opencode` for OpenCode, `~/.claude` for Claude Code. If neither is detected, the script installs the CLI only and prints a warning.

## Decision

We implement the installer as a Bash script (`install/install.sh`) that performs the full setup sequence and is safe to run any number of times.

## Consequences

The idempotent design means the installer is the canonical "fix everything" command. If a user's MCP configuration gets corrupted, they re-run the installer. If they install a second AI client after initial setup, they re-run the installer and the new client gets configured alongside the existing one. If pm is updated with new slash commands, re-running the installer copies them over. There is never a need to manually edit configuration files or figure out which step failed.

The inline `node -e` approach for JSON configuration editing is pragmatic but unusual. The script uses Node.js as a JSON processor because shell tools like `jq` may not be installed, and `sed`-based JSON manipulation is fragile and error-prone. By spawning a small Node.js program that reads, modifies, and writes the JSON file, the script gets reliable JSON handling without adding a dependency. The tradeoff is readability — inline JavaScript embedded in a Bash script is harder to read and debug than a standalone script file. Each inline script is short (about ten lines) and well-commented, which mitigates this somewhat.

The marker-based approach for agent rules (`awk` with start/end markers) is a proven pattern for idempotent file section management. It allows the pm rules to coexist with other content in the AGENTS.md file without interfering. The markers are plain markdown headings (`# PM Autonomous Filing Rules` and `# END PM Autonomous Filing Rules`), so they are human-readable and do not look like hidden metadata. The `awk` script strips the section between markers and a subsequent `cat` appends the fresh version from `install/agents-rules.md`. This means changes to the agents rules source file are automatically picked up on the next install run.

Client detection by directory existence is simple but imperfect. A user might have `~/.config/opencode` from a previous installation that is no longer active, or might have `~/.claude` from the Claude desktop app rather than Claude Code. The script does not verify that the detected client is actually installed and functional — it only checks for the configuration directory. In practice, false positives are harmless (the script writes configuration files that the inactive client will never read), but they could confuse users who see "OpenCode: true" in the detection output for a client they do not use.

The script does not support uninstallation. There is no `install.sh --uninstall` that removes the MCP registration, deletes slash commands, and strips agent rules. Uninstalling requires manual cleanup: `npm uninstall -g project-management`, then editing each client's configuration to remove the pm-tools entries. For a developer tool, this is an acceptable gap, but it means that abandoned pm installations leave artifacts in client configuration files.

One platform consideration: the script uses `set -euo pipefail` for safety (exit on error, undefined variable errors, and pipeline failures), and relies on standard Unix utilities (`cp`, `mkdir`, `awk`, `grep`, `mktemp`). It is tested on Linux and macOS. Windows is not supported — the script relies on Unix paths, shebangs, and shell semantics. Windows users would need WSL or a manual setup process.
