# ADR 009: Slash Commands as Markdown Prompt Files

## Status

Accepted

## Context

pm needs to provide guided, multi-step workflows for AI agents — things like "pick the next highest-priority story and work on it," "create a new project with proper structure," or "audit the current project for issues." These workflows involve sequences of pm CLI invocations interspersed with reasoning, file reading, code editing, and decision-making. They are too complex for a single tool call but too structured to leave entirely to the agent's discretion.

One approach would be to implement these workflows as code — TypeScript functions or shell scripts that orchestrate multiple steps programmatically. This would give precise control over the execution sequence and allow for error handling, branching logic, and state management between steps. However, the "executor" in pm's case is an AI agent, not a runtime. The agent needs instructions it can interpret and adapt, not a rigid script it must follow byte-for-byte. An AI agent is better at following natural-language instructions with decision points ("if no argument was given, identify the next highest-priority story") than executing a deterministic program.

Both OpenCode and Claude Code support "slash commands" — markdown files that are injected into the agent's context when the user types a command like `/pm-work-on`. The agent reads the markdown as instructions and follows them, using its available tools (file editing, CLI execution, search) to carry out each step. This mechanism is already built into the clients; pm does not need to implement any runtime for slash command execution.

The question was whether to use this existing mechanism or build something more structured. A more structured approach might involve a custom command runner that parses a DSL, manages state between steps, and provides guaranteed execution order. But this would duplicate functionality already available in the AI agent itself, and it would require pm to understand the agent's tool-calling interface — a coupling that would break across different clients and agent versions.

## Decision

Multi-step workflows are implemented as markdown files in `install/commands/`, one file per slash command. Each file contains natural-language instructions that describe the workflow steps, decision points, CLI commands to run, and rules to follow. These files are copied to each client's commands directory during installation (`~/.config/opencode/commands/` for OpenCode, `~/.claude/commands/` for Claude Code).

Slash commands use `$ARGUMENTS` as a placeholder for user-provided parameters. For example, `/pm-work-on PM-E001-S003` substitutes `PM-E001-S003` into the command's instructions wherever `$ARGUMENTS` appears. The commands invoke `pm` CLI subcommands directly (e.g., `pm work <STORY_CODE>`, `pm story update <STORY_CODE> --status done`) rather than using internal function calls or MCP tool invocations.

The canonical source for all slash commands lives in `install/commands/` within the pm repository. The installer copies them to client-specific locations, ensuring both OpenCode and Claude Code receive identical instructions. There is no client-specific variant of any slash command.

## Consequences

The greatest advantage of this approach is that slash commands are zero-code. Adding a new workflow requires writing a markdown file, not TypeScript. This dramatically lowers the barrier to creating new workflows and makes them accessible to anyone who can write clear instructions. The pm project currently ships eleven slash commands covering project creation, story execution, epic refinement, prioritization, auditing, and status checking — all implemented as markdown with no executable code.

Because the AI agent interprets the markdown rather than executing it mechanically, the workflows are inherently adaptive. If a step fails (say, a `pm work` command returns an error because the story code is invalid), the agent can read the error, reason about it, and either retry with a corrected code or ask the user for clarification. A hardcoded script would need explicit error handling for every failure mode; the markdown approach gets error recovery "for free" from the agent's general reasoning capabilities.

The `$ARGUMENTS` mechanism is simple but limited. It provides positional substitution without any parsing, validation, or type checking. If a user invokes `/pm-work-on` without an argument, the command must handle the missing-argument case in its instructions (and the `pm-work-on` command does — it has a "if no argument was given" branch that falls back to auto-selecting the next story). More complex parameter patterns — optional flags, multiple named parameters, default values — are not supported by the substitution mechanism. This has not been a limitation so far because the commands are designed to accept at most one argument and use the pm CLI for anything more structured.

A downside of the markdown approach is that workflows cannot be tested programmatically. There is no way to unit-test a slash command the way you would test a function — the "runtime" is an AI agent, and the output depends on the agent's interpretation. The correctness of a slash command is verified through manual use and iteration on the phrasing. If a command produces unexpected agent behavior, the fix is to rewrite the instructions more clearly, not to debug code.

The single-source design (canonical files in `install/commands/`, copied to clients by the installer) means that updating a slash command requires re-running the installer. Editing the file in `~/.config/opencode/commands/` directly would work temporarily, but the change would be overwritten by the next install. This is intentional — the installer is the deployment mechanism — but it could surprise someone who edits commands in place and later loses their changes.

Finally, because slash commands invoke `pm` CLI subcommands directly (not MCP tools), they depend on the `pm` binary being available on the PATH. This is the same dependency that the MCP server has (see ADR 006). The installer ensures this by running `npm install -g .` as its first step, but a user who installs the MCP server manually without the CLI would find that slash commands fail silently or produce confusing errors.
