# ADR 5: Stable Hierarchical Code Scheme

## Context

Every entity in the system — project, epic, story — needs a stable identifier that agents and humans can reference across sessions, across repositories, and across time. These identifiers appear in CLI commands (`pm work PM-E001-S003`), in agent conversations ("I filed PM-E003-S004"), in slash command arguments, and in the YAML data files themselves.

Several identification schemes were considered:

- **UUIDs** are globally unique and collision-free but are opaque, unreadable, and impossible to type or remember. An agent saying "I filed 8f3a2b1c-..." provides no context about what project or epic the story belongs to.

- **Auto-incrementing integers** (1, 2, 3) are simple but ambiguous — "story 3" of which epic, in which project?

- **Hierarchical codes** (`PM-E001-S003`) encode the full path from project to epic to story in a single string. They are readable, typeable, and self-documenting. The project code (`PM`) is a 2-6 letter mnemonic. Epic numbers (`E001`) and story numbers (`S001`) are zero-padded for consistent sorting.

The hierarchical approach requires that codes be immutable once assigned — renumbering would break all references. It also requires uniqueness enforcement at each level.

## Decision

We will use a hierarchical code scheme: project codes are 2-6 uppercase letters (regex `^[A-Z]{2,6}$`), epic IDs are `E{NNN}` (three-digit zero-padded), and story IDs are `S{NNN}`. Full codes are composed as `{PROJECT}-{EPIC}` for epics and `{PROJECT}-{EPIC}-{STORY}` for stories. Codes are assigned sequentially and never reused. The `nextEpicNumber()` and `nextStoryNumber()` functions in `src/lib/codes.ts` scan existing files to determine the next available number.

## Status

Accepted.

## Consequences

Codes are human-readable and self-locating. Given `PM-E001-S003`, any agent or human can immediately identify the project (PM), the epic (E001), and the story (S003). This makes CLI commands, log messages, and agent conversations clear without additional context lookups.

The zero-padding to three digits means the system supports up to 999 epics per project and 999 stories per epic. This is more than sufficient for the intended use case but is a hard ceiling. Exceeding it would require a schema change.

The sequential assignment via filesystem scanning (`nextEpicNumber` reads the epics directory and finds the maximum existing number) means that if an epic file is deleted, its number is not reused. This is intentional — codes are stable identifiers, and reuse would create confusion in historical references.

The project code uniqueness is enforced by checking for directory existence via `isProjectCodeTaken()`. Epic and story uniqueness within their parents is enforced by the sequential number assignment and the Zod `superRefine` duplicate-ID check on the epic schema.

A limitation is that project codes cannot be renamed after creation without manually renaming the directory and updating all epic and story `code` fields. There is no `pm rename` command — this is a deliberate simplification given that codes are meant to be permanent.
